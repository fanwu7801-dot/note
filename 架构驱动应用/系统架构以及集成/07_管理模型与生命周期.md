# 管理模型与生命周期

学习目标：理解为什么平台化嵌入式工程需要统一的管理模型，掌握 `register`、`lifecycle`、`manager`、`trace` 的设计方法，并知道在 FreeRTOS 和摩托车 VCU 场景下如何把这些信息整理成 AI 能理解、能辅助诊断、能辅助写代码的工程输入。

这篇文档接在“对象模型设计”和“数据模型设计”之后。

前面两节已经解决两个问题：

```text
对象模型：设备、服务、应用如何拥有统一身份。
数据模型：对象内部如何拆分 cfg / ctx / data / ops。
```

本节解决的是：

```text
这些对象创建出来以后，谁来注册，谁来初始化，谁来启动，谁来停止，谁来记录状态，谁来帮助诊断问题。
```

如果没有统一管理模型，代码很容易变成下面这样：

```c
aht21_init();
aht21_start();

imu_init();
imu_start();

bms_comm_init();
bms_comm_start();

vcu_service_init();
vcu_service_start();
```

这在小 Demo 里可以跑，但在真实 VCU 工程中会很快失控：

- 启动顺序靠人记忆，容易把依赖关系写错；
- 某个模块初始化失败后，不知道系统应该继续、重试还是停止；
- 日志散落在每个模块里，难以还原启动链路；
- FreeRTOS 任务、队列、定时器创建失败后，没有统一记录；
- AI 诊断时拿不到结构化上下文，只能猜。

管理模型的核心目标是：

> 让系统中的设备、服务、应用都按统一规则注册、初始化、启动、停止、观察和诊断。

## 1. 什么是面向切面 AOP

AOP 的全称是 `Aspect-Oriented Programming`，中文常翻译为“面向切面编程”。

对嵌入式工程来说，可以先这样理解：

> 面向切面，就是把很多模块都会用到，但又不属于某个模块核心业务的动作，放到统一入口处理。

例如温湿度设备关心的是读取温湿度：

```c
read_temperature();
read_humidity();
```

电池通信模块关心的是解析 BMS 报文：

```c
parse_bms_voltage();
parse_bms_current();
parse_bms_soc();
```

VCU 控制服务关心的是状态机和扭矩仲裁：

```c
check_vehicle_inputs();
update_vcu_state();
calculate_torque_request();
```

这些都是各自的主要业务。

但工程里还有一类动作，几乎所有模块都需要：

- 参数检查；
- 生命周期状态检查；
- 初始化、启动、停止；
- 错误码转换；
- 日志记录；
- 运行耗时统计；
- 诊断事件上报；
- FreeRTOS 任务状态观察。

这些动作不属于 AHT21、BMS、VCU 某一个模块的核心业务，而是系统级管理动作。

不推荐每个模块手写：

```c
aht21_init();
aht21_log_status();

bms_comm_init();
bms_comm_log_status();

vcu_service_init();
vcu_service_log_status();
```

更推荐放到统一入口：

```c
platform_object_register(&g_aht21.base.object);
platform_object_register(&g_bms_comm.base.object);
platform_object_register(&g_vcu_service.base.object);

platform_manager_init_all();
platform_manager_start_all();
platform_trace_lifecycle(obj, action, ret, cost_us);
```

这就是本节的主线：

```text
对象自己实现业务。
Manager 统一编排生命周期。
Trace 统一观察过程。
AI 诊断基于结构化 trace 和快照分析问题。
```

## 2. 小白先理解四个关键词

| 关键词 | 小白理解 | 工程含义 |
| --- | --- | --- |
| `register` | 报名登记 | 把对象加入管理表，让系统知道它存在 |
| `lifecycle` | 生命周期 | 对象从创建、初始化、启动、运行、停止到错误的状态流转 |
| `manager` | 管理员 | 统一调用对象的生命周期接口，并维护顺序和依赖 |
| `trace` | 过程记录 | 记录谁在什么时候做了什么，结果如何，耗时多少 |

可以把系统启动想象成学校开学：

```text
register：学生先登记进名单。
lifecycle：学生从未报到 -> 已报到 -> 上课中 -> 放学。
manager：班主任按名单点名、安排上课。
trace：考勤表记录谁迟到、谁缺席、什么时候发生。
```

放到嵌入式里就是：

```text
register：设备、服务注册到表里。
lifecycle：对象状态从 CREATED -> INITIALIZED -> STARTED。
manager：设备管理器、服务管理器、系统管理器统一执行。
trace：记录初始化和启动结果，供日志、诊断、AI 分析使用。
```

## 3. 生命周期状态设计

统一生命周期状态建议放在 `platform_object_t` 中。

```c
typedef enum
{
    PLATFORM_OBJECT_STATE_CREATED = 0,
    PLATFORM_OBJECT_STATE_REGISTERED,
    PLATFORM_OBJECT_STATE_INITIALIZED,
    PLATFORM_OBJECT_STATE_STARTED,
    PLATFORM_OBJECT_STATE_STOPPED,
    PLATFORM_OBJECT_STATE_ERROR,
    PLATFORM_OBJECT_STATE_DESTROYED,
} platform_object_state_t;
```

推荐状态流转：

```text
CREATED
    -> register()
REGISTERED
    -> init()
INITIALIZED
    -> start()
STARTED
    -> stop()
STOPPED
    -> deinit()
DESTROYED
```

异常时进入：

```text
任意关键阶段失败 -> ERROR
```

如果前面公共错误码里还没有状态错误，建议补充一个统一错误码：

```c
PLATFORM_ERR_INVALID_STATE = -16
```

也可以先复用 `PLATFORM_ERR_NOT_READY`，但在生命周期管理里，`INVALID_STATE` 表达会更清楚：它说明“当前对象状态不允许执行这个动作”。

### 3.1 lifecycle 和 status 不要混在一起

初学者很容易把生命周期和运行状态混在一个枚举里。

错误示例：

```c
typedef enum
{
    DEVICE_READY,
    DEVICE_ONLINE,
    DEVICE_STARTED,
    DEVICE_SLEEP,
    DEVICE_ERROR,
} device_state_t;
```

这里 `STARTED` 是生命周期，`ONLINE` 是健康状态，`SLEEP` 是电源状态，混在一起后会很难维护。

推荐拆开：

```text
object.state      管生命周期：创建、注册、初始化、启动、停止
device.status     管健康状态：在线、离线、忙、错误
device.power      管电源状态：上电、睡眠、挂起
service.status    管服务运行状态：空闲、运行、阻塞、错误
```

对 VCU 来说还要再区分：

```text
VCU 状态机：OFF、READY、DRIVE、CHARGE、FAULT
对象生命周期：VCU 服务是否已初始化、是否已启动
故障状态：是否存在油门故障、BMS 超时、电机故障
```

这几个不是一回事。

## 4. register：为什么需要注册

`register` 的作用是把对象放进系统管理表。

如果不注册，管理器就不知道这个对象存在，也无法统一初始化、启动、停止、打印状态或诊断。

### 4.1 推荐注册接口

```c
typedef struct
{
    platform_object_t *items[PLATFORM_OBJECT_MAX_COUNT];
    uint16_t count;
} platform_object_registry_t;

Platform_ErrorCode_t platform_object_register(
    platform_object_registry_t *registry,
    platform_object_t *obj)
{
    PLATFORM_RETURN_IF_NULL(registry);
    PLATFORM_RETURN_IF_NULL(obj);

    if (registry->count >= PLATFORM_OBJECT_MAX_COUNT)
    {
        return PLATFORM_ERR_NO_RESOURCE;
    }

    if (obj->state != PLATFORM_OBJECT_STATE_CREATED)
    {
        return PLATFORM_ERR_INVALID_STATE;
    }

    registry->items[registry->count] = obj;
    registry->count++;

    obj->state = PLATFORM_OBJECT_STATE_REGISTERED;

    platform_trace_object_registered(obj);
    return PLATFORM_OK;
}
```

这段代码做了几件事：

1. 检查注册表和对象指针是否合法。
2. 检查注册表容量是否足够。
3. 检查对象是否处于允许注册的状态。
4. 把对象指针放进表里。
5. 更新对象生命周期状态。
6. 记录 trace。

### 4.2 静态注册和动态注册

嵌入式工程更推荐静态注册。

```c
static platform_device_t *g_device_table[] = {
    &g_adc_throttle.base,
    &g_can_motor.base,
    &g_can_bms.base,
};

static platform_service_t *g_service_table[] = {
    &g_log_service.base,
    &g_vcu_service.base,
    &g_diag_service.base,
};
```

静态注册的优点：

- 不依赖动态内存；
- 启动顺序可审查；
- 链接后对象数量可控；
- 适合安全相关和资源受限系统。

动态注册适合插件化、热插拔或复杂上位机程序。摩托车 VCU 一般不建议把关键控制对象做成运行时随意动态注册。

## 5. lifecycle：统一生命周期入口

生命周期函数表建议统一定义：

```c
typedef enum
{
    PLATFORM_LIFECYCLE_INIT = 0,
    PLATFORM_LIFECYCLE_START,
    PLATFORM_LIFECYCLE_STOP,
    PLATFORM_LIFECYCLE_DEINIT,
} platform_lifecycle_action_t;

typedef struct
{
    Platform_ErrorCode_t (*init)(void *self);
    Platform_ErrorCode_t (*start)(void *self);
    Platform_ErrorCode_t (*stop)(void *self);
    Platform_ErrorCode_t (*deinit)(void *self);
} platform_lifecycle_ops_t;
```

不同对象实现自己的函数：

```c
static Platform_ErrorCode_t vcu_service_init(void *ctx);
static Platform_ErrorCode_t vcu_service_start(void *ctx);
static Platform_ErrorCode_t vcu_service_stop(void *ctx);
```

统一入口负责检查状态、调用函数、更新状态、记录 trace：

```c
Platform_ErrorCode_t platform_lifecycle_run(
    platform_object_t *obj,
    const platform_lifecycle_ops_t *ops,
    void *self,
    platform_lifecycle_action_t action)
{
    Platform_ErrorCode_t ret = PLATFORM_OK;
    uint32_t start_us;
    uint32_t cost_us;

    PLATFORM_RETURN_IF_NULL(obj);
    PLATFORM_RETURN_IF_NULL(ops);
    PLATFORM_RETURN_IF_NULL(self);

    start_us = platform_time_get_us();
    platform_trace_lifecycle_enter(obj, action);

    switch (action)
    {
        case PLATFORM_LIFECYCLE_INIT:
            if (obj->state != PLATFORM_OBJECT_STATE_REGISTERED)
            {
                ret = PLATFORM_ERR_INVALID_STATE;
            }
            else if (ops->init != PLATFORM_NULL)
            {
                ret = ops->init(self);
            }
            if (ret == PLATFORM_OK)
            {
                obj->state = PLATFORM_OBJECT_STATE_INITIALIZED;
            }
            break;

        case PLATFORM_LIFECYCLE_START:
            if (obj->state != PLATFORM_OBJECT_STATE_INITIALIZED &&
                obj->state != PLATFORM_OBJECT_STATE_STOPPED)
            {
                ret = PLATFORM_ERR_INVALID_STATE;
            }
            else if (ops->start != PLATFORM_NULL)
            {
                ret = ops->start(self);
            }
            if (ret == PLATFORM_OK)
            {
                obj->state = PLATFORM_OBJECT_STATE_STARTED;
            }
            break;

        case PLATFORM_LIFECYCLE_STOP:
            if (obj->state != PLATFORM_OBJECT_STATE_STARTED)
            {
                ret = PLATFORM_ERR_INVALID_STATE;
            }
            else if (ops->stop != PLATFORM_NULL)
            {
                ret = ops->stop(self);
            }
            if (ret == PLATFORM_OK)
            {
                obj->state = PLATFORM_OBJECT_STATE_STOPPED;
            }
            break;

        case PLATFORM_LIFECYCLE_DEINIT:
            if (ops->deinit != PLATFORM_NULL)
            {
                ret = ops->deinit(self);
            }
            if (ret == PLATFORM_OK)
            {
                obj->state = PLATFORM_OBJECT_STATE_DESTROYED;
            }
            break;

        default:
            ret = PLATFORM_ERR_INVALID_PARAM;
            break;
    }

    if (ret != PLATFORM_OK)
    {
        obj->state = PLATFORM_OBJECT_STATE_ERROR;
    }

    cost_us = platform_time_get_us() - start_us;
    platform_trace_lifecycle_exit(obj, action, ret, cost_us);

    return ret;
}
```

小白要抓住重点：

> 具体模块只写“自己怎么初始化、怎么启动”。公共入口负责“能不能调用、调用后状态怎么变、失败后怎么记录”。

## 6. 三大 Manager 的职责分工

平台化工程里建议至少有三类管理器：

```text
ObjectManager   管统一对象身份和注册表
DeviceManager   管硬件设备生命周期
ServiceManager  管系统服务生命周期
```

在 VCU 工程中还可能增加：

```text
TaskManager      管 FreeRTOS 任务和周期调度
DiagManager      管故障码、诊断快照和 UDS 数据
PowerManager     管上下电和低功耗
```

初学阶段先掌握三大 Manager。

### 6.1 ObjectManager

`ObjectManager` 负责所有对象的公共管理。

典型职责：

- 注册对象；
- 按名称查找对象；
- 按类型遍历对象；
- 打印对象状态；
- 给 trace 和 AI 诊断提供对象基础信息。

示例接口：

```c
Platform_ErrorCode_t platform_object_manager_register(platform_object_t *obj);
platform_object_t *platform_object_manager_find(const char *name);
void platform_object_manager_dump_all(void);
```

它不应该直接操作 GPIO、CAN、ADC，也不应该直接计算 VCU 扭矩。

### 6.2 DeviceManager

`DeviceManager` 负责硬件设备对象。

典型对象：

- 油门 ADC；
- 刹车输入；
- 电机 CAN 通信；
- BMS CAN 通信；
- EEPROM/Flash；
- IMU；
- 蜂鸣器、继电器、LED。

典型职责：

- 统一初始化所有设备；
- 按依赖顺序启动设备；
- 停止或休眠设备；
- 查询设备健康状态；
- 记录设备生命周期 trace。

示例：

```c
Platform_ErrorCode_t platform_device_manager_init_all(void)
{
    uint16_t i;

    for (i = 0u; i < g_device_count; i++)
    {
        PLATFORM_RETURN_IF_ERROR(platform_device_init(g_device_table[i]));
    }

    return PLATFORM_OK;
}
```

### 6.3 ServiceManager

`ServiceManager` 负责系统服务对象。

典型服务：

- 日志服务；
- CAN 通信服务；
- VCU 控制服务；
- 诊断服务；
- OTA 服务；
- 数据记录服务。

服务通常依赖设备，因此启动顺序一般是：

```text
platform_common 初始化
    -> device_manager init/start
    -> service_manager init/start
    -> app 或控制任务启动
```

示例：

```c
Platform_ErrorCode_t platform_system_start(void)
{
    PLATFORM_RETURN_IF_ERROR(platform_device_manager_init_all());
    PLATFORM_RETURN_IF_ERROR(platform_device_manager_start_all());

    PLATFORM_RETURN_IF_ERROR(platform_service_manager_init_all());
    PLATFORM_RETURN_IF_ERROR(platform_service_manager_start_all());

    return PLATFORM_OK;
}
```

## 7. 摩托车 VCU 的启动链路

摩托车 VCU 不是普通传感器，它是控制中心。

推荐启动链路：

```text
1. platform_common_init()
   初始化基础类型、错误码、日志后端、时间基准。

2. platform_mcu_init()
   初始化时钟、NVIC、看门狗、基础外设。

3. platform_device_manager_init_all()
   初始化 ADC、CAN、Flash、IO、传感器等设备。

4. platform_device_manager_start_all()
   启动设备通信和采样能力。

5. platform_service_manager_init_all()
   初始化日志、通信、诊断、VCU 控制服务。

6. platform_service_manager_start_all()
   创建队列、事件组、FreeRTOS 任务或启动周期调度。

7. vTaskStartScheduler()
   进入 FreeRTOS 调度。
```

### 7.1 VCU 控制服务的生命周期

```text
CREATED
    -> REGISTERED
    -> init：加载配置、清空上下文、初始化状态机、校验标定
    -> start：创建控制任务、注册 CAN 接收回调、启动周期定时器
    -> STARTED
```

示例：

```c
static Platform_ErrorCode_t vcu_service_init(void *ctx)
{
    vcu_t *self = (vcu_t *)ctx;

    PLATFORM_RETURN_IF_NULL(self);
    PLATFORM_RETURN_IF_NULL(self->cfg);

    vcu_ctx_reset(&self->ctx);
    vcu_data_reset(&self->data);
    vcu_fault_manager_reset(&self->faults);

    self->ctx.state = VCU_STATE_OFF;
    return vcu_cfg_validate(self->cfg);
}
```

```c
static Platform_ErrorCode_t vcu_service_start(void *ctx)
{
    vcu_t *self = (vcu_t *)ctx;

    PLATFORM_RETURN_IF_NULL(self);

    PLATFORM_RETURN_IF_ERROR(vcu_can_register_rx_callback(self));
    PLATFORM_RETURN_IF_ERROR(vcu_task_create(self));

    self->ctx.state = VCU_STATE_READY;
    return PLATFORM_OK;
}
```

注意：

> `lifecycle state = STARTED` 只说明 VCU 服务已经启动；`VCU_STATE_READY/DRIVE/FAULT` 是业务状态机，两者不要混淆。

## 8. FreeRTOS trace：为什么需要它

有些同学会把它写成 “FreeRots trace”，更准确的写法是 **FreeRTOS trace**。

FreeRTOS trace 的目的不是“多打印日志”，而是观察 RTOS 运行时行为。

在 VCU 工程里，很多问题不是业务代码一眼能看出来的，而是和任务调度有关：

- 10 ms 控制任务是否真的每 10 ms 运行；
- CAN 接收任务是否被低优先级任务阻塞；
- 诊断任务是否占用 CPU 太久；
- 某个队列是否频繁满；
- 某个互斥锁是否导致优先级反转；
- 看门狗喂狗任务是否被饿死；
- 任务栈是否接近溢出。

这些问题需要 FreeRTOS trace 帮你还原现场。

### 8.1 FreeRTOS trace 可以观察什么

| 观察项 | 说明 | VCU 价值 |
| --- | --- | --- |
| 任务切换 | 谁切到谁，什么时候切换 | 判断控制任务是否被抢占过久 |
| 任务执行时间 | 每个任务运行多久 | 发现超时任务和 CPU 占用高的任务 |
| 队列发送/接收 | 队列是否满、是否阻塞 | 判断 CAN/诊断数据是否堆积 |
| 信号量/互斥锁 | 是否等待锁，等待多久 | 判断共享数据保护是否影响实时性 |
| 中断进入/退出 | 中断频率和耗时 | 判断 CAN/ADC 中断是否过重 |
| 栈高水位 | 任务剩余栈空间 | 提前发现栈风险 |

### 8.2 FreeRTOS 内置 trace hook

FreeRTOS 提供了一些 trace 宏，可以在 `FreeRTOSConfig.h` 中打开和重定义。

常见宏：

```c
#define traceTASK_SWITCHED_IN()
#define traceTASK_SWITCHED_OUT()
#define traceQUEUE_SEND(pxQueue)
#define traceQUEUE_RECEIVE(pxQueue)
#define traceTASK_CREATE(pxNewTCB)
#define traceTASK_DELETE(pxTaskToDelete)
```

你可以把它们接到平台 trace 模块：

```c
#define traceTASK_SWITCHED_IN() \
    platform_trace_task_switched_in(pxCurrentTCB)

#define traceQUEUE_SEND(pxQueue) \
    platform_trace_queue_send(pxQueue)
```

实际工程里不同 FreeRTOS 版本和移植层对内部变量可见性不同。初学阶段不要一上来就读 TCB 私有字段，先记录可拿到的信息，例如任务名、tick、事件类型、队列句柄。

### 8.3 最小可落地的 RTOS trace 事件

可以先定义一个轻量事件结构：

```c
typedef enum
{
    PLATFORM_TRACE_EVT_LIFECYCLE = 0,
    PLATFORM_TRACE_EVT_TASK_SWITCH_IN,
    PLATFORM_TRACE_EVT_TASK_SWITCH_OUT,
    PLATFORM_TRACE_EVT_QUEUE_SEND,
    PLATFORM_TRACE_EVT_QUEUE_RECEIVE,
    PLATFORM_TRACE_EVT_QUEUE_FULL,
    PLATFORM_TRACE_EVT_STACK_LOW,
} platform_trace_event_type_t;

typedef struct
{
    uint32_t timestamp_us;
    platform_trace_event_type_t type;
    const char *name;
    uint32_t arg0;
    uint32_t arg1;
    Platform_ErrorCode_t ret;
} platform_trace_event_t;
```

记录到环形缓冲区：

```c
#define PLATFORM_TRACE_BUFFER_SIZE 128u

static platform_trace_event_t g_trace_buffer[PLATFORM_TRACE_BUFFER_SIZE];
static volatile uint16_t g_trace_write_index;

void platform_trace_push(const platform_trace_event_t *event)
{
    uint16_t index;

    if (event == PLATFORM_NULL)
    {
        return;
    }

    index = g_trace_write_index % PLATFORM_TRACE_BUFFER_SIZE;
    g_trace_buffer[index] = *event;
    g_trace_write_index++;
}
```

注意：如果在中断或 trace hook 里写缓冲区，需要考虑临界区、原子性和重入问题。初学阶段可以先只在任务上下文记录生命周期事件，再逐步扩展 RTOS trace。

### 8.4 VCU 里最有价值的 FreeRTOS trace

建议优先观察这些任务：

```text
Task_VCU_10ms       VCU 控制主任务
Task_CAN_Rx         CAN 接收解析任务
Task_CAN_Tx         CAN 发送任务
Task_Diag_100ms     诊断任务
Task_Log            日志任务
Task_Watchdog       看门狗任务
```

关键检查：

```text
Task_VCU_10ms 是否周期稳定。
Task_CAN_Rx 是否出现队列满。
Task_Diag_100ms 是否影响控制任务。
Task_Log 是否因为打印过慢阻塞高优先级任务。
Task_Watchdog 是否能够按周期运行。
```

示例 trace：

```text
t=10000us task=Task_VCU_10ms event=switch_in
t=10420us task=Task_VCU_10ms event=switch_out cost=420us
t=10500us queue=CAN_RX event=receive count=3
t=20000us task=Task_VCU_10ms event=switch_in
t=21650us task=Task_VCU_10ms event=switch_out cost=1650us warn=over_budget
```

如果 10 ms 控制任务预算是 2 ms，第二次运行还可以接受；如果某次跑到 8 ms，就必须分析原因。

## 9. Trace 和普通 log 的区别

很多初学者会问：trace 和 log 不都是打印吗？

不是。

| 项目 | log | trace |
| --- | --- | --- |
| 目的 | 给人看运行信息 | 还原系统行为过程 |
| 内容 | 可读文本 | 结构化事件 |
| 频率 | 不宜太高 | 可以高频，但要轻量 |
| 位置 | 业务关键点 | 生命周期、调度、队列、状态变化 |
| 消费者 | 工程师 | 工程师、脚本、AI 诊断工具 |

不推荐：

```c
printf("vcu start ok\r\n");
```

更推荐：

```c
platform_trace_lifecycle_exit(
    &self->base.object,
    PLATFORM_LIFECYCLE_START,
    PLATFORM_OK,
    cost_us);
```

打印只是 trace 的一种输出方式。trace 更重要的是结构化数据。

## 11. 设计代码时应该怎么做

初学阶段建议按下面顺序落地，不要一口气做完整平台。

### 11.1 第一步：先做最小对象和生命周期

最小文件：

```text
platform_object.h
platform_lifecycle.h
platform_lifecycle.c
```

先实现：

- 对象状态枚举；
- 生命周期动作枚举；
- 生命周期函数表；
- `platform_lifecycle_run()`；
- trace 空实现或弱实现。

### 11.2 第二步：再做设备和服务管理器

文件：

```text
platform_device_manager.h
platform_device_manager.c
platform_service_manager.h
platform_service_manager.c
```

先支持：

- 静态表；
- `init_all()`；
- `start_all()`；
- `stop_all()`；
- `dump_all()`。

### 11.3 第三步：接入一个真实对象

不要一开始接所有模块。

先接一个简单设备，例如 LED 或 CAN 通信对象：

```text
确认它能注册。
确认它能 init/start。
确认失败时状态进入 ERROR。
确认 trace 能看到它的生命周期事件。
```

再接 VCU Service。

### 11.4 第四步：增加 FreeRTOS trace

先不要追求完整可视化。

最小目标：

```text
记录任务名、事件类型、tick、耗时。
记录队列满和队列接收数量。
记录 VCU 10 ms 任务是否超预算。
```

### 11.5 第五步：让 AI 基于 trace 诊断

把 trace 输出整理成结构化文本或 CSV：

```text
timestamp_us,type,name,arg0,arg1,ret
1000,LIFECYCLE,vcu_service,START,0,OK
10000,TASK_IN,Task_VCU_10ms,0,0,OK
10420,TASK_OUT,Task_VCU_10ms,420,0,OK
```

再交给 AI 分析。

## 12. 常见错误

### 错误 1：注册和初始化混在一起

错误：

```c
platform_device_register(dev)
{
    dev->init();
}
```

注册只是加入名单。初始化是生命周期动作。两者要分开，便于控制顺序和诊断失败点。

### 错误 2：Manager 里写具体业务

错误：

```c
device_manager_start_all()
{
    aht21_start();
    bms_parse_soc();
    vcu_calculate_torque();
}
```

Manager 只负责编排生命周期，不负责具体业务计算。

### 错误 3：生命周期失败后继续启动

如果电机 CAN 设备启动失败，VCU 控制服务可能不应该进入正常启动。至少要进入降级或故障路径，不能假装成功。

### 错误 4：trace 里只存字符串

字符串日志方便看，但不方便统计和 AI 分析。关键事件应有结构化字段：时间、类型、对象、动作、返回值、耗时。

### 错误 5：在 trace hook 里做重活

FreeRTOS trace hook 可能在高频路径上执行，不要在里面 `printf`、阻塞等待、申请内存或做复杂格式化。优先写入轻量环形缓冲区。

### 错误 6：AI 诊断输入太少

错误提示：

```text
我的 VCU 不工作，帮我看看。
```

正确方式：

```text
给 AI：对象状态、任务表、trace、故障码、最近修改、复现步骤。
```

## 13. Q&A

### Q1：为什么不直接在 main 里按顺序调用所有 init？

小项目可以这样做。但项目变大后，启动顺序、失败处理、状态记录和低功耗都会变复杂。Manager 的价值是把这些系统级动作统一起来。

### Q2：register 是不是多余？我直接有全局变量不行吗？

全局变量只能说明对象存在，不能说明它进入了系统管理。注册之后，系统才能统一遍历、查找、启动、停止和诊断。

### Q3：生命周期状态是不是越多越好？

不是。状态越多，状态机越复杂。初学阶段先用 `CREATED / REGISTERED / INITIALIZED / STARTED / STOPPED / ERROR` 就够用。

### Q4：FreeRTOS trace 一定要用专业工具吗？

不一定。Tracealyzer 这类工具很强，但初学阶段可以先用轻量环形缓冲区记录关键事件。先解决“能看到关键事实”，再考虑可视化。

### Q5：trace 会不会影响实时性？

会，所以 trace 必须轻量。高频 hook 里不要打印，不要阻塞，不要动态分配内存。只记录必要字段，并控制缓冲区大小。

### Q6：VCU 控制任务需要 trace 哪些信息？

优先记录周期开始时间、结束时间、耗时、输入质量、状态机状态、故障掩码、扭矩输出和是否超预算。

### Q7：AI 能不能直接帮我判断故障原因？

可以辅助，但前提是你提供足够事实。AI 更适合根据 trace 和快照提出排查路径，不应该代替工程师直接下安全结论。

### Q8：为什么说 AI 诊断需要结构化输入？

因为结构化输入能减少歧义。`t=10000us task=VCU cost=420us ret=OK` 比“好像有点卡”更容易分析。

### Q9：Manager 会不会让代码变复杂？

一开始会多几个文件，但长期会降低复杂度。因为新增设备或服务时，你只需要实现自己的生命周期函数并加入表，而不是改一堆启动代码。

### Q10：量产 VCU 可以直接用本文示例吗？

不可以直接照搬。本文示例用于学习架构思想。量产 VCU 还需要完整需求、故障分析、诊断协议、安全策略、测试证据、代码审查和车辆级验证。

## 14. 设计检查清单

写完管理模型后，用下面清单检查：

1. 每个对象是否都有统一 `name/type/state`？
2. 注册、初始化、启动、停止是否分开？
3. Manager 是否只做生命周期编排，不写具体业务？
4. 设备是否先于依赖它的服务启动？
5. 生命周期失败是否能进入 `ERROR` 并记录错误码？
6. Trace 是否记录对象名、动作、返回值、耗时？
7. FreeRTOS trace 是否避免在 hook 中做阻塞操作？
8. VCU 10 ms 控制任务是否有周期和耗时记录？
9. AI 诊断是否能拿到对象状态、任务表、trace 和故障快照？
10. 新增一个设备时，是否只需要实现生命周期函数并加入静态表？

## 15. 最终记忆

```text
register 解决“系统知不知道你”。
lifecycle 解决“你现在处于哪个阶段”。
manager 解决“谁按顺序管理你”。
trace 解决“发生过什么、结果如何、耗时多少”。
FreeRTOS trace 解决“任务、队列、调度有没有影响实时性”。
AI 诊断解决“基于事实整理可能原因和排查路径”。
```

一句话总结：

> 管理模型不是为了把代码写得复杂，而是为了让设备、服务、VCU 控制逻辑在统一生命周期下可启动、可停止、可观察、可诊断、可交给 AI 辅助分析。

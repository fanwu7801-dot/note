# 数据模型设计：cfg / ctx / data / ops

学习目标：把一个可运行的设备、服务或 VCU 功能，拆成边界清晰的四类数据：**配置（cfg）**、**运行上下文（ctx）**、**业务快照（data）**、**操作接口（ops）**。这不是数学上的“四元数”，而是嵌入式软件中常用的对象数据模型。

本节承接“对象模型设计”。上一节解决的是对象的身份、类型和生命周期；本节解决的是：一个对象内部哪些内容是固定规则、哪些是运行控制信息、哪些是对外数据、哪些是行为入口。

---

## 1. 为什么需要四元组

小工程里常见下面这种写法：

```c
typedef struct
{
    uint8_t initialized;
    uint8_t i2c_addr;
    uint32_t sample_period_ms;
    float temperature;
    float humidity;
    Platform_ErrorCode_t (*read)(void *self);
} aht21_t;
```

它能运行，但把不同生命周期、不同访问权限的数据混在了一起：

- 板级地址和采样周期是配置，不应被业务代码随意改写；
- `initialized`、重试次数、接收状态是驱动内部运行过程；
- 温湿度是上层需要读取的结果快照；
- 函数指针是行为，不是状态。

当对象变成电机控制器、BMS 通信或整车 VCU 功能时，混放会导致配置被误写、跨任务读取半更新数据、故障状态与业务数据混淆，最终难以诊断和维护。

推荐的对象形态如下：

```text
对象 = base + cfg + ctx + data + ops

cfg   固定规则：硬件、车型变型、标定、阈值、周期
ctx   内部过程：状态机、计时器、通信收发、重试、锁
data  对外事实：输入、计算结果、命令、诊断快照
ops   行为入口：初始化、周期执行、接收处理、命令控制
```

四部分不是按“字段长得像不像”划分，而是按以下三个问题划分：

1. 谁能修改它？
2. 它在编译/标定时确定，还是随运行过程变化？
3. 它描述内部执行过程，还是描述需要对外发布的业务事实？

---

## 2. 四元组的边界

| 部分 | 核心问题 | 典型内容 | 修改者 |
| --- | --- | --- | --- |
| `cfg` | 系统应按什么规则工作？ | 引脚、CAN ID、周期、限值、车型参数 | 编译配置、标定管理、受控诊断服务 |
| `ctx` | 当前执行到哪里、内部是否可继续？ | 初始化状态、状态机、超时、计数器、通信会话 | 本模块实现 |
| `data` | 当前可被其他模块使用的事实是什么？ | 信号快照、计算结果、命令快照、故障快照 | 本模块写，上层/订阅者读 |
| `ops` | 外部能请求哪些动作？ | `init`、`run_10ms`、`on_can_rx`、`set_mode` | 静态函数表 |

一个重要边界是：**`ctx` 不等于“所有会变化的数据”**。

例如 AHT21 的“上次 I2C 事务是否超时”属于 `ctx`；“温度为 25.3 °C、该值何时更新、是否有效”属于 `data`。前者是驱动实现细节，后者是供上层判断和使用的业务结果。

### 2.1 推荐的 C 语言组织方式

继承自上一节的对象模型时，基类必须放在第一个字段。`cfg` 和 `ops` 通常由静态常量实例持有，因此对象中更适合保存 `const` 指针，而不是复制一份：

```c
typedef struct aht21 aht21_t;

typedef struct
{
    uint8_t bus_id;
    uint8_t i2c_addr;
    uint32_t sample_period_ms;
} aht21_cfg_t;

typedef struct
{
    uint8_t initialized;
    uint8_t transfer_pending;
    uint32_t last_sample_tick_ms;
    uint16_t consecutive_error_count;
} aht21_ctx_t;

typedef struct
{
    uint32_t timestamp_ms;
    uint8_t valid;
    float temperature_deg_c;
    float humidity_percent;
} aht21_data_t;

typedef struct
{
    Platform_ErrorCode_t (*init)(aht21_t *self);
    Platform_ErrorCode_t (*sample)(aht21_t *self);
    Platform_ErrorCode_t (*get_snapshot)(const aht21_t *self,
                                         aht21_data_t *snapshot);
    Platform_ErrorCode_t (*sleep)(aht21_t *self);
    Platform_ErrorCode_t (*wake)(aht21_t *self);
} aht21_ops_t;

struct aht21
{
    platform_device_t base;       /* 必须放第一个字段 */
    const aht21_cfg_t *cfg;       /* 静态配置/标定 */
    aht21_ctx_t ctx;              /* 私有运行上下文 */
    aht21_data_t data;            /* 对外数据快照 */
    const aht21_ops_t *ops;       /* 静态行为表 */
};
```

这里的 `const` 表达的是访问约束，不代表数据一定存放在 Flash。若配置需要在线标定，应由标定服务完成校验、版本管理和原子切换；不能让普通业务模块直接写 `cfg`。

## 3. cfg：静态配置与受控标定

`cfg` 放“对象如何被部署和约束”的数据。它应该稳定、可追溯、可审查。

适合放入 `cfg`：

- 硬件资源：CAN 通道、GPIO、ADC 通道、地址、DMA 资源；
- 功能周期和超时：`10 ms` 控制周期、`100 ms` 信号超时；
- 产品变型：两轮/三轮车型、是否具备倒车、仪表协议版本；
- 标定量：扭矩上限、斜率、温度降额曲线、轮径、速比；
- 安全策略参数：降级阈值、故障去抖时间、默认安全状态。

不适合放入 `cfg`：当前车速、当前档位、接收计数器、故障是否已发生、上次报文时间。

```c
typedef struct vcu vcu_t;

typedef struct
{
    uint32_t can_id_motor_status;
    uint32_t can_id_bms_status;
    uint16_t task_period_ms;
    uint16_t motor_msg_timeout_ms;
    uint16_t bms_msg_timeout_ms;

    int16_t max_drive_torque_nm_x10;
    int16_t max_regen_torque_nm_x10;
    uint16_t throttle_plausibility_max_permille;
    uint16_t wheel_circumference_mm;
} vcu_cfg_t;
```

对于 VCU，建议把“产品配置”和“可在线标定参数”拆成两个子结构或两个版本对象。前者由软件版本控制，后者必须有 CRC、版本号、范围校验与回滚策略。

---

## 4. ctx：模块私有的运行上下文

`ctx` 用于让模块正确执行，而不是供其他模块直接依赖。它通常包含：

- 生命周期与状态机状态；
- 周期调度时间戳、超时监控、去抖和重试计数；
- CAN/I2C/SPI 等通信收发过程状态；
- 互斥锁、环形队列索引、异步事务句柄；
- 内部故障判定中间量。

```c
typedef enum
{
    VCU_STATE_OFF = 0,
    VCU_STATE_INIT,
    VCU_STATE_READY,
    VCU_STATE_DRIVE,
    VCU_STATE_CHARGE,
    VCU_STATE_FAULT,
} vcu_state_t;

typedef struct
{
    vcu_state_t state;
    uint8_t initialized;
    uint8_t motor_status_rx_seen;
    uint8_t bms_status_rx_seen;
    uint32_t motor_status_last_rx_ms;
    uint32_t bms_status_last_rx_ms;
    uint16_t throttle_fault_debounce_count;
    uint16_t watchdog_kick_count;
} vcu_ctx_t;
```

`ctx` 应由该模块的 `.c` 文件私有，或至少不向应用层暴露可写指针。若为了调试需要查看，提供只读诊断快照接口；不要让其他模块直接改状态机状态。

---

## 5. data：可消费的业务快照

`data` 是对外契约。上层、诊断、记录和通信模块看到的应是明确单位、明确有效性、明确时间基准的快照。

每个关键数据建议至少具备：

- 值和单位；固定点量纲优先于控制路径中的 `float`；
- `valid` 或 `quality`，区分有效、超时、无效、降级；
- 时间戳或数据年龄，支持新鲜度判断；
- 来源或序号，必要时支持一致性与调试追踪。

```c
typedef enum
{
    VCU_SIGNAL_INVALID = 0,
    VCU_SIGNAL_VALID,
    VCU_SIGNAL_TIMEOUT,
    VCU_SIGNAL_DEGRADED,
} vcu_signal_quality_t;

typedef struct
{
    uint32_t timestamp_ms;
    uint16_t sequence;
    vcu_signal_quality_t quality;
} vcu_sample_meta_t;

typedef struct
{
    vcu_sample_meta_t meta;
    uint16_t throttle_permille;
    uint16_t front_brake_permille;
    uint16_t rear_brake_permille;
    int16_t vehicle_speed_kph_x100;
    int16_t motor_speed_rpm;
    int16_t battery_voltage_v_x10;
    int16_t battery_current_a_x10;
    uint16_t soc_permille;
} vcu_input_data_t;

typedef struct
{
    vcu_sample_meta_t meta;
    int16_t requested_drive_torque_nm_x10;
    int16_t requested_regen_torque_nm_x10;
    uint8_t drive_enable;
    uint8_t charge_enable;
} vcu_command_data_t;

typedef struct
{
    vcu_sample_meta_t meta;
    uint32_t active_fault_mask;
    uint32_t latched_fault_mask;
    vcu_state_t state;
} vcu_diag_data_t;

typedef struct
{
    vcu_input_data_t input;
    vcu_command_data_t command;
    vcu_diag_data_t diag;
} vcu_data_t;
```

上例有意把 **输入、输出命令、诊断** 分开。把它们混成一个“车辆状态”结构体，常会让接收报文覆盖控制命令，或让诊断读取到不同周期拼出的不一致状态。

### 5.1 并发与一致性

VCU 中 CAN 接收中断、10 ms 控制任务、100 ms 诊断任务可能同时访问数据。不能假设结构体整体读写天然原子。

推荐规则：

1. 中断只解析报文并写入受保护的输入缓冲区，复杂控制计算放到周期任务。
2. 控制任务先取得一致输入快照，再计算，再一次性发布命令快照。
3. 跨任务读取用临界区、互斥锁、双缓冲或序列锁；方案取决于 MCU、RTOS 和访问频率。
4. 读取者检查 `quality` 和数据年龄，超时数据不得继续参与扭矩请求。

可提供快照接口，而不是暴露内部可写对象：

```c
Platform_ErrorCode_t vcu_get_data_snapshot(const vcu_t *self,
                                           vcu_data_t *snapshot);
```

## 6. ops：行为接口，而不是万能 read/write

`ops` 让对象可以被统一管理，但接口必须体现对象语义。传感器适合 `sample()`；VCU 不适合仅用无类型的 `read()`/`write()`，因为“写入一个数据”无法表达它是驾驶请求、标定更新，还是诊断清故障。

```c
typedef struct
{
    Platform_ErrorCode_t (*init)(vcu_t *self);
    Platform_ErrorCode_t (*start)(vcu_t *self);
    Platform_ErrorCode_t (*run_10ms)(vcu_t *self, uint32_t now_ms);
    Platform_ErrorCode_t (*run_100ms)(vcu_t *self, uint32_t now_ms);
    Platform_ErrorCode_t (*on_can_rx)(vcu_t *self,
                                      uint32_t can_id,
                                      const uint8_t *payload,
                                      uint8_t dlc,
                                      uint32_t now_ms);
    Platform_ErrorCode_t (*request_mode)(vcu_t *self, vcu_state_t target);
    Platform_ErrorCode_t (*get_snapshot)(const vcu_t *self, vcu_data_t *out);
    Platform_ErrorCode_t (*clear_non_latched_faults)(vcu_t *self);
    Platform_ErrorCode_t (*stop)(vcu_t *self);
} vcu_ops_t;
```

统一原则：

- 第一个参数是具体对象 `self`，而不是没有类型信息的全局变量；
- 返回统一的 `Platform_ErrorCode_t`；
- `ops` 本身不保存运行值，所有状态通过 `self->cfg/ctx/data` 访问；
- 安全相关操作使用明确、可审计的命名，例如 `request_mode()`、`clear_non_latched_faults()`；
- `ops` 表通常为 `static const`，禁止运行时替换。

## 7. 摩托车 VCU 如何落地这个四元组

摩托车 VCU 不应被当成单一传感器驱动。它是车辆控制功能对象：接收油门、刹车、BMS、电机控制器和车身输入，执行状态机与扭矩仲裁，向电机控制器、仪表、BMS 发布命令和状态。

推荐的 VCU 对象如下：

```c
struct vcu
{
    platform_service_t base;      /* VCU 是控制服务，非单纯硬件设备 */
    const vcu_cfg_t *cfg;
    vcu_ctx_t ctx;
    vcu_data_t data;
    const vcu_ops_t *ops;
};
```

| 四元组 | 摩托车 VCU 的典型内容 | 不应放入 |
| --- | --- | --- |
| `cfg` | CAN ID、超时、轮径、扭矩/回收上限、温度降额、车型功能开关 | 当前 SOC、当前车速、当前故障 |
| `ctx` | VCU 状态机、报文接收时间、去抖计数、握手阶段、任务内部标志 | 对外承诺的车辆状态快照 |
| `data` | 油门/刹车/车速/BMS/电机输入，扭矩请求，运行模式，故障快照 | CAN 接收 FIFO 索引、局部临时变量 |
| `ops` | 初始化、CAN 接收、10 ms 控制、100 ms 诊断、模式请求、快照读取 | 任意模块都能直接写扭矩 |

### 7.1 推荐运行链路

```text
CAN/ADC 输入
    -> on_can_rx() / 采样适配层
    -> data.input（带时间戳和质量）
    -> run_10ms()
    -> 合理性检查 + 状态机 + 扭矩仲裁 + 限扭/降额
    -> data.command + data.diag
    -> CAN 发送适配层发布给电机控制器/仪表
```

控制任务的关键顺序应固定：

```text
刷新输入有效性
    -> 检查超时与油门/刹车合理性
    -> 更新故障与 VCU 状态机
    -> 计算基础扭矩请求
    -> 叠加 BMS、电机、温度、模式等限值
    -> 不满足使能条件时输出安全扭矩（通常为 0）
    -> 原子发布命令与诊断快照
```

这里的“安全扭矩”必须由整车安全目标、动力系统接口和故障策略定义；本文不假定某一具体车型的故障降级值。

### 7.2 一个最小的 10 ms 控制框架

```c
static Platform_ErrorCode_t vcu_run_10ms(vcu_t *self, uint32_t now_ms)
{
    vcu_data_t snapshot;

    PLATFORM_RETURN_IF_NULL(self);
    PLATFORM_RETURN_IF_ERROR(vcu_get_data_snapshot(self, &snapshot));

    vcu_update_signal_quality(self, now_ms);
    vcu_update_faults(self, &snapshot, now_ms);
    vcu_update_state_machine(self, &snapshot);

    if (self->ctx.state == VCU_STATE_DRIVE &&
        self->data.diag.active_fault_mask == 0u)
    {
        vcu_calculate_torque_command(self, &snapshot);
    }
    else
    {
        self->data.command.requested_drive_torque_nm_x10 = 0;
        self->data.command.requested_regen_torque_nm_x10 = 0;
        self->data.command.drive_enable = 0u;
    }

    vcu_publish_command_snapshot(self, now_ms);
    return PLATFORM_OK;
}
```

示例强调边界，不是可直接用于量产车辆的安全软件。量产 VCU 还需根据安全目标建立需求追踪、故障分析、独立监控、看门狗策略、通信 E2E/计数器和相应的验证证据。

## 8. 常见错误

### 错误 1：把运行状态都塞进 `data`

`rx_fifo_index`、互斥锁、状态机中间步骤属于 `ctx`。它们不应成为其他模块的依赖。

### 错误 2：把实时车速、SOC 放进 `cfg`

`cfg` 是规则，`data` 是事实。实时信号必须带有效性和时效信息。

### 错误 3：外部模块直接写 `data.command`

这样会绕过状态机、限值和故障仲裁。外部只能提出类型明确的请求，最终命令由 VCU 控制任务统一生成。

### 错误 4：所有接口都是 `read()` / `write()`

对 VCU 来说这会掩盖意图。优先使用 `on_can_rx()`、`run_10ms()`、`request_mode()`、`get_snapshot()` 这样的业务接口。

### 错误 5：只有 `valid`，没有时间戳和质量状态

一个报文“曾经有效”不代表现在仍可用。至少应支持 `VALID`、`TIMEOUT`、`INVALID`，并基于数据年龄做控制决策。

## 9. 设计检查清单

1. 每个字段能否明确归属 `cfg`、`ctx` 或 `data`？
2. `cfg` 是否由 `const` 指针保护，并具备版本/校验策略？
3. `ctx` 是否不向业务层暴露可写访问？
4. 每个关键输入和输出是否有单位、时间戳和质量状态？
5. 跨中断/任务的数据是否通过一致性快照读取？
6. VCU 命令是否只由统一控制周期发布，而非被外部直接写入？
7. `ops` 是否使用了表达业务意图的强类型接口？
8. 故障、超时和降级是否进入明确的状态机和命令仲裁链路？

最终记忆：

```text
cfg 定规则，ctx 管过程，data 发布事实，ops 提供动作。
VCU 的核心不是“保存很多信号”，而是让信号的有效性、状态机、故障和扭矩命令在固定边界内可追溯地流动。
```

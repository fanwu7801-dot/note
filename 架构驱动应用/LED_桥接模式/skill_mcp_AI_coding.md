# Skill 怎么写：从理解到上手

这篇笔记的目标不是只给你一个定义，而是让你最后能自己写出一个能复用的 `skill`。

---

## 1. 什么是 Skill

`skill` 可以理解成：

> 给 AI 的一个“专项工作说明书”。

它不是普通提示词，也不是一段零散经验，而是一套可以重复调用的规则、步骤和边界。

当 AI 遇到某类任务时，加载对应的 `skill`，就能按固定方式工作，而不是每次都重新猜你的习惯。

你可以把它类比成：

- `prompt`：一句临时口头要求
- `skill`：一份可复用的岗位 SOP

所以，`skill` 的核心价值不是“让 AI 更聪明”，而是：

- 让 AI 的输出更稳定
- 让某类任务可以复用
- 让你的经验沉淀下来
- 让 AI 知道“什么时候做、怎么做、做到什么程度”

---

## 2. Skill 的作用

如果你只给 AI 一个问题，比如：

> 帮我分析这段代码

AI 往往会自由发挥，结果可能每次都不一样。

但如果你给它一个 `skill`，例如：

- 先看目录结构
- 再找主入口
- 再找关键数据流
- 最后输出“模块职责、调用链、风险点、改进建议”

那么 AI 的行为就被“工程化”了。

### skill 解决的本质问题

`skill` 主要解决 3 件事：

1. 解决“怎么做”
   把步骤固定下来，避免 AI 每次临场发挥。

2. 解决“做多深”
   明确输出粒度，比如只做概览、还是做到能修改代码。

3. 解决“什么时候用”
   指定触发条件，避免乱用。

---

## 3. 一个好 Skill 应该包含什么

一个能用的 `skill`，通常至少要有下面 5 部分：

### 3.1 这个 skill 是干什么的

先写清楚用途。

例如：

> 用于分析嵌入式项目中的 LED 控制链路，定位从应用层到驱动层的调用关系。

这部分回答的是：

- 解决什么问题
- 面向什么场景
- 不负责什么

### 3.2 什么时候触发

也就是使用条件。

例如：

- 当用户提到 “LED”
- 当用户要分析驱动调用链
- 当用户要定位某个外设的分层设计

这部分很重要，因为没有触发条件，`skill` 会变成一堆“好建议”，但不具备自动化价值。

### 3.3 执行步骤

这是最核心的部分。

不要写空话，比如：

- 仔细分析代码
- 全面理解架构

这种话对 AI 没约束。

应该写成可执行步骤，例如：

1. 先找 LED 相关文件名、宏定义、结构体和入口函数
2. 找出应用层调用 LED 的起点
3. 追踪到中间抽象层
4. 再追踪到底层驱动
5. 输出分层关系和调用时序

### 3.4 输出格式

告诉 AI 最后应该怎么交付结果。

例如：

- 先给结论
- 再给调用链
- 再给关键文件路径
- 最后给风险点

没有输出格式，结果容易漂。

### 3.5 边界和限制

告诉 AI 不要做什么。

例如：

- 不要猜测未看到的代码行为
- 找不到实现时要明确说明
- 不要直接修改文件，除非用户要求

这部分决定结果是否可靠。

---

## 4. Skill 的本质结构

你可以把一个 `skill` 理解成下面这个公式：

```text
Skill = 使用场景 + 触发条件 + 执行步骤 + 输出格式 + 边界限制
```

如果缺一项，效果都会变差：

- 没有使用场景：不知道它是干嘛的
- 没有触发条件：不知道何时用
- 没有执行步骤：AI 还是自由发挥
- 没有输出格式：结果不稳定
- 没有边界限制：容易胡说或越界

## 5. 一个最小可用 Demo

下面给你一个“分析嵌入式 LED 调用链”的 skill demo。

这个 demo 不是最复杂的，但结构是完整的，你可以直接照着学。

### Demo：`embedded-led-analysis`

```md
# embedded-led-analysis

## Purpose
用于分析嵌入式项目中 LED 相关代码的分层结构、调用链和职责边界。

## Use When
- 用户要求分析 LED 模块
- 用户要求梳理从应用层到驱动层的调用过程
- 用户需要定位 LED 控制逻辑所在文件

## Workflow
1. 先搜索文件名、宏、结构体、函数名中与 LED 相关的标识。
2. 找到应用层触发 LED 行为的入口函数。
3. 沿调用链向下追踪到中间抽象层。
4. 继续追踪到板级支持层或驱动层。
5. 标记每一层的职责：
   - 应用层：业务触发
   - 抽象层：接口适配
   - 驱动层：GPIO/PWM/寄存器控制
6. 输出关键文件、关键函数、调用顺序和设计模式特征。

## Output
按以下结构输出：
1. 一句话结论
2. 模块分层说明
3. 调用链
4. 关键文件列表
5. 风险点或改进建议

## Constraints
- 不要假设未读取到的代码内容。
- 如果调用链中断，明确指出断点位置。
- 如果存在多个 LED 实现，分别说明，不要混为一谈。
```

---

## 6. 这个 Demo 为什么是对的

你要学的不是“抄一个 demo”，而是理解它为什么这么写。

### 第一层：它先定义任务范围

`Purpose` 说明这个 skill 只负责 LED 分析，不负责所有嵌入式问题。

这样 AI 不容易跑偏。

### 第二层：它说明触发时机

`Use When` 是“什么时候该调用它”。

这能让 skill 真正可复用，而不是一篇静态笔记。

### 第三层：它把动作拆成步骤

`Workflow` 不是概念描述，而是执行顺序。

这会直接影响 AI 的稳定性。

### 第四层：它约束输出

`Output` 统一了交付结果。

这样你每次看到的回答结构基本一致，更适合工程协作。

### 第五层：它限制乱猜

`Constraints` 是质量控制。

这部分很像代码里的“异常处理”或“防呆设计”。

## 7. 你应该怎么自己写 Skill

如果你的目标是“学会自己写”，建议按下面 4 步走。

### 第 1 步：先定一个非常小的场景

不要一开始就写：

> 分析所有嵌入式代码

这个范围太大，skill 会很虚。

正确做法是先缩小：

- 分析 LED 调用链
- 分析按键扫描流程
- 分析 UART 初始化过程
- 分析任务调度入口

skill 越聚焦，越容易写好。

### 第 2 步：先写“动作”，再写“描述”

很多人写 skill 时，容易先写一堆介绍：

- 要全面分析
- 要深入理解
- 要注意架构

这些都太空。

正确做法是先列出动作：

1. 先搜什么
2. 再看什么
3. 再追踪什么
4. 最后输出什么

动作清楚了，skill 才能执行。

### 第 3 步：输出格式一定要固定

如果你想要稳定结果，就不要让 AI 自己决定怎么回答。

你应该明确要求：

- 先结论
- 后证据
- 再文件
- 再建议

这相当于你在定义“接口协议”。

### 第 4 步：加上“不要做什么”

一个好 skill 不只是教 AI 做什么，还要告诉它不能怎么做。

例如：

- 不要跳步
- 不要猜代码
- 不要在未确认时下结论
- 不要修改文件，除非明确要求

这会显著提升质量。

## 8. 一个通用模板

以后你自己写 skill，可以先套这个模板：

```md
# skill-name

## Purpose
这个 skill 用来解决什么问题。

## Use When
- 在什么场景下使用
- 用户提到什么关键词时使用
- 哪类任务适合它

## Workflow
1. 第一步做什么
2. 第二步做什么
3. 第三步做什么
4. 最终怎么收敛结果

## Output
按什么结构输出结果。

## Constraints
- 不要做什么
- 什么情况下要明确说明不确定
- 哪些行为必须避免
```

## 9. 你现在可以怎么练

如果你要真正学会，建议不要停留在“看懂”。

你可以立刻练这 3 个题目：

1. 写一个 `key-scan-analysis` skill
   用来分析按键扫描流程。

2. 写一个 `uart-init-analysis` skill
   用来分析串口初始化路径。

3. 写一个 `driver-layer-review` skill
   用来检查驱动层接口是否清晰、是否混入业务逻辑。

这 3 个题目都比“写一个万能 skill”更适合入门。

## 10. 结论

你可以把 `skill` 记成一句话：

> `skill` 不是“提示词变长了”，而是“把某类任务的方法固定下来”。

所以写 `skill` 的关键，不是文笔好，而是：

- 场景要小
- 步骤要清楚
- 输出要固定
- 边界要明确

如果你按这个思路写，skill 就会从“描述型笔记”变成“可执行 SOP”。

---

## 11. 最后给你的建议

你这个文档如果是为了“自己学习怎么写 skill”，那最好的写法不是只讲概念，而是采用下面这个结构：

1. 先解释 `skill` 是什么
2. 再解释 `skill` 有什么作用
3. 再给一个最小 demo
4. 再拆解 demo 为什么这样写
5. 最后给一个你自己套用的模板

也就是说，这篇文档的定位最好是：

> 一篇“教我自己写 skill”的入门教程

而不是：

> 一篇只介绍概念的说明文

这样你以后回头看，才能真的照着写出来。

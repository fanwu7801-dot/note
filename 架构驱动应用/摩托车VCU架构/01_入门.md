# 摩托车 VCU 架构入门

本文面向刚接触车载嵌入式软件的读者，说明摩托车 VCU 软件通常由哪些部分组成，以及应当怎样阅读一个实际工程。

文中的工程组织参考 `wybt_dcu-main-fuel` 项目的 `README.md`：该项目使用 **FreeRTOS**，README 的硬件平台项列出主控为 **GD32E513VET6**、蓝牙 SoC 为 **FR8016HA**；工程使用 CMake 构建，并在应用、RTE、BSP、中间件和驱动目录中组织源文件。

注意：README 的产物名是 `JFTS_DCU_Fuel`。这说明它是一个具体 DCU/Fuel 软件工程；仅凭 README 不能断言其已经覆盖完整摩托车 VCU 的全部功能。因此，本文借它学习“车载控制软件如何分层”，不把项目中未明确的功能当作既定事实。

## 1. 先理解：VCU 是什么

VCU 是 **Vehicle Control Unit，整车控制器**。它不直接等于电机控制器，也不只是一个 CAN 网关。

以电动摩托车为例，VCU 常需要接收油门、刹车、档位、车速、电池管理系统（BMS）和电机控制器的信息，然后根据车辆状态决定：

- 是否允许上高压或进入可骑行状态；
- 请求多少驱动扭矩或能量回收扭矩；
- 电池、电机或通信异常时怎样限扭或禁止驱动；
- 向仪表、BMS、电机控制器发送什么状态和命令；
- 怎样保存、读取和清除诊断故障。

燃油摩托车、混动摩托车或专用 DCU 的具体控制对象会不同，但软件的基本问题一致：**采集输入、判断状态、执行控制、处理故障、通过通信发布结果。**

---

## 2. 不要把“参考 AUTOSAR”理解成“必须使用 AUTOSAR”

汽车领域常用 AUTOSAR 的分层思想。小型摩托车控制器或已有项目不一定采用完整 AUTOSAR，也不需要为了分层而引入整套 AUTOSAR 工具链。

真正需要保留的是它的工程原则：

```text
业务控制逻辑不直接操作寄存器。
硬件差异集中在底层。
应用通过稳定接口读取输入、发布请求和获取诊断状态。
```

参考项目的 README 已经体现出类似分层：ASW、RTE、BSP/驱动、RTOS、UDS。可以按下面的简化结构理解：

```text
ASW / 应用功能
        ↓ 通过接口读写
RTE / 服务与接口层
        ↓ 调用
BSP、驱动、中间件、FreeRTOS
        ↓
GD32 MCU、CAN、ADC、GPIO、Flash、蓝牙等硬件
```

这张图描述的是依赖方向，而不是要求每层只能有一个目录。

## 3. 各层到底做什么

### 3.1 ASW：车辆功能逻辑

ASW（Application Software）是“车辆应该怎么工作”的代码。README 指出，该项目的 ASW 逻辑代码由 Simulink 模型生成。

对初学者来说，可以把 ASW 理解为做决策的一层：

```text
输入：油门开度、刹车状态、BMS 允许功率、电机状态
处理：状态机、合理性检查、限值计算、扭矩仲裁
输出：驱动使能、扭矩请求、故障状态、仪表请求
```

ASW 应该写成“意图”，而不是“硬件操作”。例如：

```c
if (battery_voltage_low == true)
{
    vcu_request_warning_lamp(true);
}
```

这段代码表达的是“欠压时请求点亮警告灯”。它不应知道警告灯连接到哪个 GPIO，也不应直接写 GPIO 寄存器。

### 3.2 RTE：应用与底层之间的接口

RTE（Runtime Environment）可以理解为应用层和底层软件之间的“翻译层”。完整 AUTOSAR 中 RTE 通常自动生成；在许多小型或自研项目中，它可能是手写接口、服务模块或管理器。

它的价值是让 ASW 不依赖硬件细节：

```c
Platform_ErrorCode_t rte_read_battery_voltage(int16_t *voltage_v_x10);
Platform_ErrorCode_t rte_write_motor_torque_request(int16_t torque_nm_x10);
```

ASW 只知道“读取已经换算好的电池电压”和“发布扭矩请求”。至于数据来自 ADC 还是 CAN、是否滤波、CAN ID 是多少，应该由 RTE 以下的模块负责。

### 3.3 BSP 与驱动：让软件能操作板子

BSP（Board Support Package）与驱动负责硬件相关工作，例如：

- 配置时钟、GPIO、中断和启动顺序；
- 操作 CAN、ADC、PWM、Flash、串口等外设；
- 按实际原理图完成引脚映射和电压/电流换算；
- 提供板级下载、调试和 OpenOCD 配置。

README 中将板级驱动标为 `BSP_DRV`，底层驱动抽象标为 `MCU_DRV`。初学时可以先记住：**BSP 更关心这块板子怎么连接；MCU 驱动更关心 MCU 外设怎么使用。**

### 3.4 FreeRTOS：让多个周期任务有序运行

README 明确该项目使用 FreeRTOS。它通常负责任务、消息队列、互斥锁、软件定时器等运行机制。

VCU 常见的任务划分可能是：

| 周期/事件 | 典型工作 |
| --- | --- |
| CAN 接收中断或接收任务 | 收报文、检查长度、更新时间戳、通知上层 |
| 10 ms 控制任务 | 输入有效性检查、状态机、扭矩和限扭计算 |
| 100 ms 诊断任务 | 故障去抖、DTC 管理、诊断状态更新 |
| 通信发送任务 | 按周期发送电机、仪表或诊断报文 |

这些周期只是常见示例，不是 README 对该项目的具体调度承诺。实际周期要以需求、CPU 负载和控制对象的实时性为准。

### 3.5 UDS：诊断通信

README 列出 `UDS` 模块。UDS（Unified Diagnostic Services）是车辆诊断协议，常用于读取故障码、读取数据、执行例程、软件升级等。

初学阶段先区分两件事：

- **CAN** 是通信总线，负责把报文传出去；
- **UDS** 是运行在诊断通信之上的服务协议，规定“读取故障码、清除故障码”等请求的含义。

因此，业务控制不能把“收到一帧 CAN 报文”直接当成“UDS 功能已经完成”。CAN 驱动、传输层、UDS 服务和诊断数据管理各有边界。

## 4. 一条信号如何从硬件走到控制逻辑

以“电池电压参与欠压保护”为例：

```text
电压采样引脚 / BMS CAN 报文
        ↓
ADC 或 CAN 驱动：获取原始数据
        ↓
BSP/信号适配：换算单位、检查范围、记录时间戳
        ↓
RTE/服务接口：提供统一的电池电压快照
        ↓
ASW：判断是否欠压，决定是否限扭、告警或禁止驱动
        ↓
RTE/通信适配：发布仪表状态或电机扭矩请求
```

这条链路最重要的不是层数，而是每一步都清楚回答：

1. 数据来自哪里？
2. 当前值是否可信、是否超时？
3. 单位是什么？
4. 谁可以修改它？
5. 结果最终发给谁？

## 5. 与 `cfg / ctx / data / ops` 的关系

上一节“数据模型设计”给出了一种非常适合 VCU 的对象划分：

```text
cfg   固定规则：CAN ID、周期、扭矩上限、阈值、车型配置
ctx   运行过程：状态机、报文超时、去抖计数、任务内部状态
data  对外快照：油门、车速、BMS 输入、扭矩请求、故障状态
ops   行为接口：初始化、周期运行、CAN 接收、读取快照、模式请求
```

例如，CAN 报文超时 `100 ms` 是规则，应在 `cfg`；“上一次收到 BMS 报文的时间”是运行过程，应在 `ctx`；“当前 BMS 数据是否超时”是上层需要知道的事实，应在 `data`；“处理一帧 CAN 数据”的函数属于 `ops`。

这个划分能避免一个常见错误：把 CAN 接收缓存、控制状态机、车辆信号和外部命令全塞进一个巨大结构体，让模块之间可以随意改数据。

## 6. 如何阅读参考工程

建议按“先运行、再跟链路、最后改功能”的顺序阅读。

### 第一步：确认构建和调试入口

README 给出的 CMake 构建方式是：

```bat
cmake --preset gcc-arm-debug
cmake --build --preset firmware
```

README 描述的构建产物名称为：

- `JFTS_DCU_Fuel.elf`：用于调试的可执行文件；
- `JFTS_DCU_Fuel.bin`：常见的烧录镜像；
- `JFTS_DCU_Fuel.hex`：另一种常见烧录格式；
- `JFTS_DCU_Fuel.map`：链接映射，用于分析代码和 RAM/Flash 占用。

但查看当前工程的 `CMakeLists.txt` 后，还应知道两个容易混淆的事实：

- 当前 CMake 项目名是 `JFTS_DCU_Hybrid`，实际输出目录被设置为 `build\cmake\output`；
- 构建产物的最终名称应以本次 CMake 配置和构建日志为准，而不能只依赖 README 中较早的 `JFTS_DCU_Fuel` 名称。

这不是小问题。烧录脚本、调试配置和 CI 如果仍引用旧 ELF 名称，就可能找不到文件。阅读或修改工程时，应先执行一次构建，再确认 `build\cmake\output` 中实际生成的 `.elf`、`.bin`、`.hex` 与 `.map`。

README 还提供 `syntax` 目标，可用于检查可移植 UDS 核心的语法：

```bat
cmake --build build\cmake --target syntax
```

调试使用 Cortex-Debug、OpenOCD 和 J-Link SWD；对应配置文件为 `BSP/Firmware/openocd_gdlink.cfg`。这些是项目实际 README 给出的入口，初学者应先确保它们可用，再分析业务逻辑。

### 第二步：从启动入口找到任务

从 `main`、板级启动文件或 FreeRTOS 任务创建处开始，回答：

- 创建了哪些任务？它们的周期或触发条件是什么？
- CAN 接收数据先进入哪个模块？
- ASW 在哪个任务或函数中运行？
- UDS 请求在哪里被分发？

不要一开始就搜索“扭矩”然后修改。先找到数据是怎样进入、被谁调用、最后怎样发出的，才能避免改错层。

### 第三步：选一条完整链路追踪

优先选择容易观察的功能，例如 CAN 信号、一个 ADC 采样量或一个诊断 DID：

```text
驱动初始化 → 数据接收/采样 → 信号换算 → RTE 接口 → ASW 使用 → CAN/诊断输出
```

能完整走通一条链路后，再学习状态机、故障管理和扭矩控制，效率会高很多。

## 7. 初学者最容易混淆的概念

| 概念 | 简单理解 | 不应混同为 |
| --- | --- | --- |
| VCU | 做整车控制决策的控制器或软件功能 | 单纯 CAN 网关或电机控制器 |
| DCU | 某个领域控制单元，范围取决于项目定义 | 必然等于完整 VCU |
| ASW | 车辆功能与算法逻辑 | 直接访问寄存器的代码 |
| RTE | 应用和基础软件之间的稳定接口/服务层 | 必然是完整 AUTOSAR 自动生成代码 |
| BSP | 与具体板卡、引脚和启动相关的软件 | 所有业务算法 |
| FreeRTOS | 任务调度和资源管理机制 | 车辆控制策略本身 |
| UDS | 诊断服务协议 | CAN 驱动本身 |

## 8. 下一步学习顺序

1. 先完成一次构建，知道 `.elf`、`.bin`、`.map` 分别是什么。
2. 找到 FreeRTOS 任务创建处，画出任务与周期。
3. 选一条 CAN 或 ADC 信号走完整数据链路。
4. 阅读 `cfg / ctx / data / ops` 数据模型，把信号的配置、过程、快照和接口分开。
5. 再进入 VCU 状态机、扭矩仲裁、故障诊断和通信超时等控制主题。

入门阶段的目标不是马上改出一个控制功能，而是先能清楚说明：**一个输入从哪里来，经过哪些层，在哪个任务里被处理，最后影响了哪个输出。**

## 9. 用真实目录理解工程分层

不要只把目录名当作标签。下面的表把当前工程中的目录、实际文件和职责对应起来：

| 层次 | 当前工程位置 | 代表文件 | 初学者应理解的职责 |
| --- | --- | --- | --- |
| 启动入口 | `ASW/main.c` | `main()` | 控制上电顺序：硬件初始化、任务初始化、启动调度器 |
| ASW 业务协调 | `ASW/` | `app_system.c`、`app_model_adapter.c`、`app_outputs.c` | 聚合输入、运行控制模型、处理输出命令和业务状态 |
| 业务模型 | `ASW/WYBT_VCUControl.c` | `WYBT_VCUControl_initialize()` | 由模型生成的控制算法；通过适配器与手写业务代码连接 |
| RTE/硬件抽象 | `RTE/` | `dcu_hal.c`、`dcu_hal.h` | 对 ASW 提供稳定的硬件操作接口，避免 ASW 直接接触寄存器 |
| 通信服务 | `RTE/COM/` | `rte_can.c`、`BleCommService.c`、`RemoteCommService.c` | 管理 CAN、蓝牙和串口通信任务、队列和路由 |
| CAN 矩阵 | `RTE/COM/Can_Matrix/` | `can_dcu.c`、`can_dcu.h` | 把 8 字节 CAN 原始帧编解码为有名称、有物理单位的信号结构体 |
| 板级/MCU 驱动 | `BSP/Software_Driver/` | `driver_can.c`、`driver_adc.c`、`imu.c` | 初始化外设，读写 CAN、ADC、GPIO、IMU 等硬件资源 |
| 芯片固件与启动 | `BSP/Firmware/` | 启动文件、链接脚本、OpenOCD 配置 | 提供 GD32E51x 的启动、标准外设库、下载和调试支持 |
| 中间件 | `Middleware/` | FreeRTOS、`uds_core`、FlashDB、CRC、DSP_LIB | 提供任务调度、诊断、非易失存储、校验和数学运算能力 |

### 9.1 一个很实用的判断方法

当你看到一个函数时，不确定它该放哪一层，可以问：

```text
它是在描述“车辆应该做什么”？                → ASW
它是在把业务意图变成统一硬件/通信接口？        → RTE
它是在配置或操作某个芯片外设、引脚或总线？      → BSP/MCU 驱动
它是在提供可复用的通用能力？                  → Middleware
```

例如 `AppSystem_RunControlCycle()` 负责控制流程，应在 ASW；`DcuHal_CanTransmitVehicle()` 负责把帧交给车辆 CAN，应在 RTE/HAL；`can_transmit()` 这类直接操作 GD32 CAN 外设的调用，应留在驱动或固件库边界内。

## 10. 从上电到控制任务：真实启动链路

当前工程的 `ASW/main.c` 入口很清晰：

```text
main()
  ├─ DcuHal_SetGlobalIrq(0)       先关闭全局中断
  ├─ HardwareInit()               初始化硬件
  ├─ SoftwareTaskInit()           创建软件资源和静态任务
  ├─ DcuHal_SetGlobalIrq(1)       打开全局中断
  └─ vTaskStartScheduler()        交给 FreeRTOS 调度
```

### 10.1 为什么先关中断

初始化期间，CAN、UART 或 ADC 的中断如果提前触发，相关队列、任务句柄和状态对象可能还没有创建完成。先关中断、完成初始化、再开中断，可以避免“中断先到，软件还没准备好”的竞态问题。

这不代表所有项目都要长时间关闭中断。原则是：**必须在中断源启用前，先准备好它依赖的缓冲区、队列、状态和处理函数。**

### 10.2 `HardwareInit()` 做了什么

`ASW/app_board_service.c` 的 `HardwareInit()` 体现了板级启动的典型顺序：

```text
驱动基础初始化
  → 开关/按键输入
  → 主电源轨与短暂等待
  → SPI、UART、车辆 CAN、ADC
  → 软件 I2C 与 IO 扩展器
  → 风挡驱动、PWM、433 MHz、IMU、RTC
  → 中断初始化
```

这里值得学习的是依赖关系，而不是死记步骤：外设供电前不能可靠通信；IO 扩展器初始化后才能驱动挂在其上的外部器件；中断应在相关硬件和软件资源可用后再启用。

### 10.3 `SoftwareTaskInit()` 做了什么

它先创建共享 I2C 的静态互斥锁，再调用 `AppServiceManager_Init()`，最后统一创建静态 FreeRTOS 任务。项目使用 `xTaskCreateStatic()`，即任务控制块和任务栈由静态数组提供。

对车载控制器而言，这种做法的直接好处是：任务创建时的 RAM 占用可预先计算，运行过程中不依赖频繁动态分配，内存行为更容易分析。

## 11. 任务分层：不要把“任务”当成“功能层”

层次解决“谁依赖谁”；任务解决“什么时候运行”。两者是两个维度。

当前工程在 `ASW/app_task.c` 中创建了如下代表性任务：

| 任务名 | 代码入口 | 触发方式 | 主要职责 |
| --- | --- | --- | --- |
| `DCUCtrl` | `task_mode_control()` | 每 10 ms，`vTaskDelayUntil()` | 初始化并运行主控制周期 |
| `Period` | `task_periodic_service()` | 每 1 ms | 周期服务，如采样/计时类工作 |
| `Body5ms` | `task_body_5ms()` | 5 ms 周期 | 车身域相关周期处理 |
| `Scan` | `task_scan()` | 任务周期 | 按键/扫描类工作 |
| `lowfre` | `LowFrequency_Tasks()` | 每 1 s | RTC 同步、诊断复位门控等低频服务 |
| `UDSSrv`、`UDSCli`、`UDSRtn` | UDS Flash 模块任务 | 独立任务 | UDS 服务端、客户端和例程执行 |
| `CAN_Rx`、`CAN_Tx` | `rte_can.c` | 通知/队列驱动 | CAN 接收解析与发送 |

`DCUCtrl` 的核心非常直接：先调用 `AppSystem_ControlInit()`，再严格每 10 ms 调用一次 `AppSystem_RunControlCycle()`。`vTaskDelayUntil()` 的意义是保持固定节拍；它比简单的“运行完再延时 10 ms”更适合周期控制任务。

> 任务周期不是随便填的。10 ms 任务的最坏执行时间必须小于 10 ms，并保留中断、通信和其他高优先级任务的余量。否则即使逻辑正确，也会因错过周期而产生控制和通信问题。

## 12. 一次 10 ms 控制周期到底发生了什么

`AppSystem_RunControlCycle()` 是理解当前工程 ASW 的最佳入口。代码中的执行顺序如下：

```text
1. AppSystem_ConsumeInputRoutes()
       取出并分发已经到达的输入
2. AppSystem_RunModelStep()
       把输入整理给 WYBT_VCUControl 模型并执行一步
3. AppSystem_PublishModelOutputs()
       将模型结果发布为 CAN 信号、SOC 事件等
4. AppSystem_DispatchOutputCommands()
       将输出变成电源、执行器和外设命令
5. AppSystem_UpdatePowerManagement()
       更新睡眠/唤醒相关电源管理
6. AppSystem_ProcessTransientDomainState()
       处理蓝牙、胎压配对等瞬态业务
```

这就是一个控制器最重要的固定顺序：**先消费输入，再计算，最后输出。**

初学者常犯的错误是让 CAN 接收中断直接驱动执行器。这样虽然响应快，但会把“接收报文”和“控制决策”耦合在一起，难以保证输入一致性。当前工程将主要控制决策集中到 10 ms 周期，方向是合理的：一个周期内可以基于同一批输入做出可追踪的决策。

## 13. CAN 的真实数据流：中断、队列、矩阵、业务

当前工程的 CAN 接收路径不是“中断里做所有工作”，而是：

```text
CAN 接收中断
  → DcuHal_CanReceiveVehicle()  从硬件 FIFO 取出一帧
  → RouteCanRxFromIsr()         检查诊断策略和允许的 CAN ID
  → xCAN_AppRxQueue             将普通应用帧放进静态队列
  → vTaskNotifyGiveFromISR()    唤醒 CAN_Rx 任务
  → task_can_rx()               逐帧调用 DCU_ProcessRxMessage()
  → can_dcu.c                   按 CAN ID 解码并更新接收信号结构体
  → ASW 读取快照/模型适配器     将所需信号送入控制模型
```

这种设计把两件事分开：

- **中断上下文**只做必须尽快完成的取帧、入队和通知；
- **任务上下文**再做报文解析、信号更新和业务计算。

这样能缩短中断占用时间，减少对其他实时任务的影响。代价是队列可能满，因此项目也维护了接收丢帧计数。设计 VCU 时必须明确：哪些帧允许丢、哪些帧必须监控超时、丢帧后该怎样降级。

### 13.1 CAN 矩阵层为什么单独存在

`RTE/COM/Can_Matrix/can_dcu.c` 由 CAN 矩阵生成，内部维护发送与接收结构体，例如 ECU、ABS、ISG、TBOX、DCU 状态等报文。它负责：

- 将字节和位域编码/解码为工程信号；
- 保存接收信号的物理值；
- 按周期和相位调度发送报文，避免同一时刻集中发送；
- 让 ASW 使用有意义的字段名，而不是手写 `data[3] >> 2`。

因此，业务代码不应直接到 `uint8_t data[8]` 中取位。信号定义、字节序、缩放、偏移和有效性应集中在 CAN 矩阵层管理。

### 13.2 发送路径

```text
ASW / 模型输出
  → AppModelAdapter_PublishCanSignals()
  → CAN 矩阵组帧与周期调度
  → can_send_message()
  → CAN 发送队列
  → CAN_Tx 任务
  → DcuHal_CanTransmitVehicle()
  → GD32 CAN 外设
```

发送同样通过队列和任务完成。这样业务层只提交“要发送的报文”，不会在 10 ms 控制任务中长时间等待硬件发送完成。

## 14. 这个工程如何对应 `cfg / ctx / data / ops`

当前工程尚不是用一个统一的 `vcu_t` 结构体表达所有功能，但其模块已经能映射到四元组思想。以 CAN 通信为例：

| 四元组 | 在当前工程中的对应物 | 说明 |
| --- | --- | --- |
| `cfg` | CAN ID、发送周期、相位、任务优先级、队列长度 | 例如 `can_dcu.c` 的发送相位、`rte_task_config.h` 的任务配置；这些是运行规则 |
| `ctx` | 任务句柄、队列句柄、接收状态、下次发送时间、丢帧计数 | 是通信模块内部执行过程，不应被 ASW 任意改写 |
| `data` | `*_Rx_t`/`*_Tx_t` 信号结构体、模型输入输出、业务快照 | 是需要被其他模块消费的车辆信息和命令 |
| `ops` | `DCU_ProcessRxMessage()`、`can_send_message()`、`AppSystem_RunControlCycle()` | 是模块对外提供的受控行为入口 |

以 10 ms 控制域为例：任务周期和模型版本属于 `cfg`；本周期是否已初始化、临时处理状态属于 `ctx`；模型输入、模型输出、执行器命令和故障状态属于 `data`；初始化、运行一步、发布输出属于 `ops`。

如果后续要进一步重构这个工程，最有价值的方向不是马上重写所有文件，而是先为关键域建立明确的快照接口，例如：

```c
Platform_ErrorCode_t app_vehicle_get_input_snapshot(AppVehicleInput *out);
Platform_ErrorCode_t app_vehicle_get_output_snapshot(AppVehicleOutput *out);
Platform_ErrorCode_t app_vehicle_run_10ms(AppVehicleController *self);
```

这样 ASW、CAN 矩阵、诊断和执行器之间的数据归属会更清楚，也更容易加入时间戳、有效性和并发保护。

## 15. 基于本工程的建议阅读顺序

如果目标是“真正读懂这个工程”，建议按下面顺序，而不是从一个很长的业务文件硬啃：

1. `CMakeLists.txt`：确认芯片内核、工具链、源文件集合和最终产物。
2. `ASW/main.c`：掌握上电、硬件初始化、任务创建、调度器启动的总流程。
3. `ASW/app_board_service.c`：了解硬件初始化为什么有顺序。
4. `ASW/app_task.c` 与 `RTE/rte_task_config.h`：列出所有任务、周期和优先级。
5. `ASW/app_system.c`：理解 10 ms 控制周期的输入、模型、输出顺序。
6. `RTE/COM/rte_can.c`：理解 CAN 中断、队列和收发任务。
7. `RTE/COM/Can_Matrix/can_dcu.c`：理解实际报文如何转成信号。
8. `ASW/app_model_adapter.c` 与 `ASW/WYBT_VCUControl.c`：最后再看模型输入输出如何与业务层连接。

每读完一层都应画一张小图，至少写下“输入是什么、输出是什么、调用者是谁、运行在哪个任务”。这样读到模型和诊断模块时，才不会迷失在大量变量和报文 ID 中。

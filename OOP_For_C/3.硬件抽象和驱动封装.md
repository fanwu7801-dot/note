### 静态对象数组管理
🚗假设有一个10个车位的停车场，我们可以使用静态对象数组来管理这些车位的状态。
- 为什么使用静态对象数组？
  - 静态对象数组在编译时分配内存，适合车位数量固定的场景。
  - 访问速度快，适合频繁查询车位状态的需求。
  - 可以用循环结构方便地遍历和管理车位。
```c
led_t led0 = {0, OFF};
led_t led1 = {1, OFF};
led_t led2 = {2, OFF};
led_t led3 = {3, OFF};

// 定义一个静态对象数组来管理车位
led_t parking_spots[10] = {
    {0, OFF},//引脚0，初始状态OFF
    {1, OFF},//引脚1，初始状态OFF
    {2, OFF},//引脚2，初始状态OFF
    {3, OFF},//引脚3，初始状态OFF
    {4, OFF},//引脚4，初始状态OFF
    {5, OFF},
    {6, OFF},
    {7, OFF},
    {8, OFF},
    {9, OFF}
};
```
因为数组的每一个元素都是一个完整的`led_t`对象，我们可以通过索引直接访问和修改每个车位的状态，并且内存是连续分配的，访问效率高。

### 对象池管理
如果一个程序需要频繁创建和销毁对象，使用对象池可以显著提高性能，减少内存碎片化。
对象池本质上是预先分配一块内存，用于存储多个对象实例。当需要一个新对象时，从池中取出一个空闲对象；当对象不再需要时，将其返回池中，而不是销毁它。
> 对象池的优点：
- 减少频繁的内存分配和释放操作，提高性能。
- 降低内存碎片化的风险。
- 适用于需要大量短生命周期对象的场景。
- 便于统一管理对象的生命周期。
类比于停车场管理系统，可以预先创建一定数量的车位对象，当有车辆进入时，从对象池中分配一个车位对象；当车辆离开时，将车位对象归还给对象池，供下一个车辆使用。
*对象池的精髓就就是***复用**，而不是每次都创建和销毁对象。
也就是空间换时间的经典案例。
#### 数据结构设计
使用内存块+状态标记来管理对象池：

![alt text](embedded-development-note-main/OOP_For_C/image-3.png)
``` cpp
typedef struct {
    bool is_free;       // 状态标记 true表示空闲，false表示已分配 
    char  data[256] ;  // 对象数据区
} ObjectPoolItem;
// 定义对象池结构体 
typedef struct {
    ObjectPoolItem* items; // 对象池数组
    size_t capacity;       // 对象池容量
    size_t used_count;           // 已分配对象数量
} ObjectPool;

// 初始化对象池
objectPool * pool_init (int size)
{
    //1. 分配对象池结构体内存
    ObjectPool* pool = (ObjectPool*)malloc(sizeof(ObjectPool));
    if(NULL == pool)
        return NULL;
    //2. 分配对象池数组内存
    pool->items = (ObjectPoolItem*)malloc(sizeof(ObjectPoolItem) * size);
    if(NULL == pool->items)
    {
        free(pool);
        return NULL;
    }
    //3. 初始化对象池属性
    pool->capacity = size;
    pool->used_count = 0;
    //4. 初始化每个对象的状态标记
    for (int i = 0; i < size; i++)
    {
        pool->items[i].is_free = true; // 标记所有对象为空闲
    }
    return pool;
}
// 从对象池中分配一个对象
ObjectPoolItem* pool_allocate(ObjectPool* pool)
{   if(NULL == pool)
    {
        return NULL;
    }
    // 遍历对象池，寻找一个空闲对象
    for (int i = 0; i < pool->capacity; i++)
    {
        if (pool->items[i].is_free) // 找到一个空闲对象
        {
            pool->items[i].is_free = false; // 标记为已分配
            pool->used_count++;
            return &pool->items[i]; // 返回对象指针
        }
    }
    return NULL; // 没有可用对象
}
// 归还对象到对象池
void pool_deallocate(ObjectPool* pool, ObjectPoolItem* item)
{
    if(NULL == pool || NULL == item)
    {
        return;
    }
    if(obj>=pool->items && obj < pool->items + pool->capacity)
    {
        item->is_free = true; // 标记为可用
        pool->used_count--;
    }
    // 这里的返回是不需要实际释放内存的，只是标记为空闲
}
// 销毁对象池
void pool_destroy(ObjectPool* pool)
{
    if(NULL == pool)
    {
        return;
    }
    // 释放对象池数组内存
    if(pool->items)
    {
        free(pool->items);
    }
    // 释放对象池结构体内存
    free(pool);
}

```
测试代码
``` cpp
int main()
{
    // 初始化对象池，容量为5
    ObjectPool* pool = pool_init(5);
    if(NULL == pool)
    {
        printf("对象池初始化失败\n");
        return -1;
    }
    // 分配3个对象
    ObjectPoolItem* obj1 = pool_allocate(pool);
    ObjectPoolItem* obj2 = pool_allocate(pool);
    ObjectPoolItem* obj3 = pool_allocate(pool);
    printf("已分配对象数量: %zu\n", pool->used_count); // 输出: 3

    // 归还1个对象
    pool_deallocate(pool, obj2);
    printf("已分配对象数量: %zu\n", pool->used_count); // 输出: 2

    // 分配2个对象
    ObjectPoolItem* obj4 = pool_allocate(pool);
    ObjectPoolItem* obj5 = pool_allocate(pool);
    printf("已分配对象数量: %zu\n", pool->used_count); // 输出: 4

    // 尝试分配超过容量的对象
    ObjectPoolItem* obj6 = pool_allocate(pool);
    if(NULL == obj6)
    {
        printf("没有可用对象\n"); // 输出: 没有可用对象
    }

    // 销毁对象池
    pool_destroy(pool);
    return 0;
}
```

### 单例模式
就像一个公司只能有一个总经理一样，单例模式确保一个类只有一个实例，并提供一个全局访问点。
> 在作用域以内，单例对象是唯一的。只有一个实例存在

*全局唯一共享访问*
唯一实例：全局访问
单例模式解决:*static*
static的作用：程序生命周期内只存在一份，且只能在本文件内访问。
单例模式的实现通常包括以下几个步骤：
1. 私有化构造函数，防止外部直接创建实例。
2. 提供一个静态方法，用于获取唯一实例。
3. 使用静态变量存储唯一实例，确保实例只被创建一次。
4. （可选）实现线程安全，防止多线程环境下创建多个实例。
#### 单例模式示例代码
``` cpp
#include <stdio.h>
#include <stdlib.h>

typedef struct {
   int port ; // 端口号
   int pageCount ; // 页面数量
} printer_t ;

printer_t * get_Printer_Instance() {
    static printer_t * instance = NULL ; // 静态变量存储唯一实例
    static int initialized = 0 ; // 标记是否已初始化
    if (!initialized) {
        // 私有化构造函数逻辑
        initialized = 1 ;
    }
    if (instance == NULL) {
        instance = (printer_t *)malloc(sizeof(printer_t));
        instance->port = 8080 ; // 初始化端口号
        instance->pageCount = 0 ; // 初始化页面数量
    }
    return instance ;
}

int main() {
    printer_t * printer1 = get_Printer_Instance();
    printer_t * printer2 = get_Printer_Instance();

    printf("Printer1 Port: %d, PageCount: %d\n", printer1->port, printer1->pageCount);
    printf("Printer2 Port: %d, PageCount: %d\n", printer2->port, printer2->pageCount);

    // 验证两个实例是否相同
    if (printer1 == printer2) {
        printf("Both printers are the same instance.\n");
    } else {
        printf("Printers are different instances.\n");
    }

    // 释放内存
    free(printer1);
    return 0;
}
```

### 深拷贝和浅拷贝
深拷贝：创建一个新对象，并复制原对象的所有数据，包括动态分配的内存。这样，两个对象互不影响。
浅拷贝：仅复制对象的指针，两个对象共享同一块内存。修改一个对象会影响另一个对象。

#### 什么是拷贝
##### 浅拷贝示例
``` cpp
typedef struct {
    char* name; // 动态分配的字符串 指针是关键
    int age;
} Person;

person p1;
p1.name = "Alice";
p1.age = 30;

person p2 = p1; // 浅拷贝

```
浅拷贝就像复制了地址，两个对象指向同一块内存区域。
两个stack变量p1和p2，它们的name成员都指向同一个heap内存区域。如果我们修改p2.name，p1.name也会受到影响。
>问题: 假如我们释放p1.name的内存，然后再访问p2.name，就会导致未定义行为，因为p2.name指向的内存已经被释放了。此时，p2.name变成了野指针
- 意外修改
- 重复释放
- 内存泄漏(悬空指针，野指针)
##### 深拷贝示例
深拷贝:数据完全独立
1. malloc新内存
2. memcpy数据
3. 指向新内存
``` cpp
typedef struct {
    char* name; // 动态分配的字符串 指针是关键
    int age;
} Person;
person p1;
p1.name = (char*)malloc(20);
strcpy(p1.name, "Alice");
p1.age = 30;
person p2;
// 深拷贝
p2.name = (char*)malloc(20); // 分配新内存
strcpy(p2.name, p1.name); // 复制数据
p2.age = p1.age;
```
深拷贝就像复制了内容，两个对象各自拥有独立的内存区域。修改一个对象不会影响另一个对象。
- 数据独立
- 安全释放
- 内存管理简单
代价: 额外的内存分配和数据复制开销，但换来了数据的独立性和安全性。

#### 总结
什么时候用深拷贝，什么时候用浅拷贝？
- 使用浅拷贝的场景：
  - 对象较小且不包含动态分配的内存。
  - 性能要求较高，且可以确保对象生命周期内不会发生冲突。
- 使用深拷贝的场景：
  - 对象包含动态分配的内存或复杂的数据结构。
  - 需要确保对象之间的数据独立性，防止意外修改和内存管理问题。
##### 嵌入式系统中
1. 传感器数据处理：传感器数据通常是动态分配的缓冲区，使用深拷贝可以确保数据独立性，防止数据被意外修改。
2. 配置管理：系统配置通常存储在动态分配的结构体中，使用深拷贝可以确保配置数据的安全性和独立性。
3. 通信协议处理：通信数据包通常需要深拷贝以保证数据完整性，防止在多线程环境下数据被篡改或丢失。
*安全性*>*性能*
``` cpp
typedef struct {
    char* data; // 动态分配的数据缓冲区
    size_t length;
} SensorData;
// 处理传感器数据的函数，使用深拷贝
void processSensorData(SensorData* input) {
    // 深拷贝传感器数据
    SensorData localCopy;
    localCopy.length = input->length;
    localCopy.data = (char*)malloc(localCopy.length);
    if(NULL == localCopy.data)
    {
        // 处理内存分配失败
        return;
    }
    memcpy(localCopy.data, input->data, localCopy.length);

    // 处理本地副本数据
    // ...

    // 释放本地副本内存
    free(localCopy.data);
} 
// 浅拷贝
void processSensorDataShallow(SensorData* input) {
    // 浅拷贝传感器数据
    SensorData localCopy = *input; // 仅复制指针

    // 处理本地副本数据
    // ...

    // 不释放本地副本内存，因为它指向的是输入数据的内存
}
```
### 内存对齐
使用内存对齐可以提高数据访问效率，减少处理器的访问时间。
- 什么是内存对齐？
>内存对齐是指数据在内存中的存储地址按照一定的规则进行排列，以提高访问效率。
- 为什么需要内存对齐？
  - 提高访问速度：处理器通常对对齐的数据访问更快，减少访问时间。
  - 减少内存碎片：对齐可以减少内存碎片，提高内存利用率。
  - 硬件要求：某些处理器对未对齐的数据访问会产生异常或性能下降。
![alt text](embedded-development-note-main/OOP_For_C/image-4.png)
cpu的数据总线宽度决定了内存对齐的粒度。例如，32位处理器的数据总线宽度为4字节，通常要求数据按4字节对齐。
当cpu访问未对齐的数据时，可能需要进行多次内存访问，导致性能下降，甚至引发硬件异常。
例如，一个32位处理器访问一个未对齐的4字节整数，可能需要两次内存访问，而对齐的数据只需一次访问。
数据类型的大小就是对齐边界
``` cpp 
struct Example {
    char a;      // 1字节
    int b;       // 4字节
    short c;     // 2字节
};
```
这里的实际内存布局可能如下：
```Offset  Size  Member
0       1     a
1       3     （填充字节）
4       4     b
8       2     c
10      2     （填充字节）
```
也就是结构体的总大小是12字节，而不是7字节。
*优化内存*
- 调整成员顺序：将大尺寸成员放在前面，小尺寸成员放在后面，减少填充字节。
- 使用编译器指令：某些编译器提供了指令来控制内存对齐方式，如`#pragma pack`。
- 自定义对齐方式：根据具体需求，自定义数据结构的对齐方式。
``` cpp
#pragma pack(push, 1) // 设置1字节对齐
struct PackedExample {
  int a;   // 4字节
  short c; // 2字节
  char b;  // 1字节
};
#pragma pack(pop) // 恢复默认对齐方式
// 这里的实际内存布局如下：
// Offset  Size  Member
// 0       4     a
// 4       2     c
// 6       1     b
// 总大小为7字节，没有填充字节
```

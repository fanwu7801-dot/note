## 问题指南

作为嵌入式开发者，遇到问题时快速、清晰地定位并复现问题，是节省时间与保证质量的关键。本指南提供统一的上报模板、排查流程、复现路径模板以及各角色（上报者/排查者/修复者/验证者）的 checklist，便于团队协作与事后复盘。

---

### 目标

- 让问题描述可复现、可验证。
- 提供结构化的排查步骤，减少反复沟通成本。
- 确保修复后有验证步骤与回归覆盖。

---

## 一、总体处理流程（简要）

1. 接收/上报：使用统一模板上报问题。
2. 初步 triage：确认环境、是否可复现、优先级与受影响范围。
3. 复现：在本地或受控环境复现问题并收集证据（日志、dump、截图）。
4. 排查与定位：最小化复现、查看差异、使用调试工具定位根因。
5. 修复：实现修复并提交 PR，包含回归测试或复现用例。
6. 验证：QA 或 PR 审核方验证问题已被解决。
7. 关闭与复盘：写短的 RCA（根因分析）与预防措施。

## 二、如何撰写高质量的问题（上报模板）

在上报问题前，请尽量把下面的内容准备齐全——越完整，排查越快。

复现模板（可复制）:

```
标题：简短描述（产品/模块 - 简要现象）
环境：硬件型号、固件版本、OS/SDK 版本、编译器版本、构建时间、分支/提交哈希
重现概率：始终 / 偶发（约 x%）
复现步骤：按序列出最小化的复现步骤（每步写明命令或操作）
期望结果：描述期望行为
实际结果：描述实际行为（含截图/视频/错误码）
相关日志：附粘贴或上传关键日志（含时间戳）
最小复现代码/项目：附上最小项目或说明如何构建复现环境
是否可提供设备/样板机：是/否（若是，提供联系方式或访问方式）
其他信息：网络状态、功耗策略、特定配置、最近更改（最近提交）、是否热插拔/断电等
```

写问题时的要点：
- 用明确的步骤说明如何复现（“我做了 A，然后 B，结果 C”）。
- 提供版本和提交哈希，避免“最新”模糊描述。
- 贴关键日志，不要只说“报错”，要给具体错误堆栈或时间点。

## 三、排查（诊断）流程（可直接套用）

1. 确认环境与版本
	- 收集 `git rev-parse --short HEAD`、构建产物版本、设备型号、外设固件版本。
2. 重现问题
	- 在 reporter 给出的步骤上尝试重现；如果重现失败，按“最小复现”逐步简化场景。
3. 收集证据
	- 关键日志、串口日志、core dump、异常栈、网络抓包、功耗/时序波形截图。
4. 搜索与排重
	- 检查 issue 列表、PR、最近提交、CI 失败记录，看是否是已知回归或重复问题。
5. 二分查找（当怀疑回归引入）
	- 使用 `git bisect` 找到引入问题的提交范围。
6. 隔离与最小化
	- 关闭不相关模块、使用最小固件/固件配置，找到最小复现用例。
7. 使用调试工具
	- 使用 `gdb`（或 `arm-none-eabi-gdb`）、J-Link、OpenOCD、或串口单步调试定位崩溃点。
8. 并发/时序问题排查
	- 加日志、使用断言、增加延时、模拟不同调度/中断场景。
9. 验证理论与修复
	- 修改代码、增加检测或保护、重新构建并验证是否解决问题。
10. 编写回归用例
	- 将最小复现作为自动化测试或手工验证步骤纳入回归。

## 四、常用命令与工具（参考）

- 版本与差异： `git log --oneline -n 20`, `git diff`, `git bisect start` / `git bisect run`
- 搜索代码/日志： `grep -n "关键字" -R .`, `rg`（ripgrep）
- 构建相关： `make clean && make all` / `cmake --build` / `west build`（取决于项目）
- 日志与系统： `dmesg`, `journalctl -u <service>`, `tail -f <logfile>`
- 网络/抓包： `tcpdump`, `wireshark`（保存 pcap）
- 二进制分析： `arm-none-eabi-objdump -d`, `readelf -a`, `nm`, `size`
- 调试器： `gdb`, `arm-none-eabi-gdb`, `openocd`, `JLinkGDBServer` + `gdb`
- 串口工具： `minicom`, `picocom`, `PuTTY`（Windows）
- 静态/动态分析： `valgrind`（Linux）、`asan`/`ubsan`（如果支持）

提示：把关键命令和输出保存为附件或粘贴到 issue 中（用 ``` 代码块）。

## 五、Checklist（可复制到 Issue 或 PR 模板）

上报者（Reporter）Checklist：

- [ ] 已给出清晰可执行的复现步骤
- [ ] 附上固件/软件版本与提交哈希
- [ ] 附上相关日志/错误堆栈（包含时间戳）
- [ ] 给出期望结果与实际结果对比
- [ ] 提供最小复现项目或说明无法提供的原因

Triager（分配人）Checklist：

- [ ] 能否在受控环境复现？（是/否/部分）
- [ ] 优先级与严重级别已标注（P0/P1/P2）
- [ ] 是否为重复问题（如是，链接到现有 issue）
- [ ] 指定负责开发者或团队与预计 ETA

开发者（Fixer）Checklist：

- [ ] 本地或 CI 上能稳定复现
- [ ] 找到可能根因并写出定位步骤
- [ ] 提交修复 PR，包含最小回归用例或自动化测试
- [ ] 在本地/CI 运行全部相关测试通过
- [ ] PR 描述包含复现步骤与验证方法

验证（QA）Checklist：

- [ ] 按 PR 中给出的验证步骤验证问题已修复
- [ ] 在受影响的不同环境/配置上进行验证（至少一台实际设备）
- [ ] 验证回归测试未引入新问题

## 六、根因分析（RCA）模板（简洁版）

```
问题摘要：一句话总结发生了什么
影响范围：受影响的功能/设备/用户数/版本
发生时间：首次发现与最后出现的时间
重现概率：始终/偶发
根因：技术上导致问题的直接原因
外部/间接因素：配置、环境、第三方库、硬件等
检测方式：如何被发现（监控、用户上报、CI）
修复与缓解：采取了哪些操作修复或临时缓解
后续措施：防止复发的长期方案（测试/监控/流程改进）
负责人：
关闭条件：验证通过、回归测试通过、部署完成
```

## 七、快速排查技巧（经验总结）

- 先确认是否可复现：不可复现的“问题”通常是沟通或环境差异造成的。
- 把问题最小化到可控场景：越小的复现用例越容易定位。
- 经常保存对比构建或二进制，方便回滚与 `git bisect`。
- 对嵌入式平台，多关注时序、资源（内存/堆栈）和外设初始化顺序。
- 使用日志等级（DEBUG/INFO/WARN/ERROR）并在关键点打时间戳。

## 设计模式
在c++/python以及java当中，这些面对对象的语言都常用设计模式来解决众多的系统架构问题。而作为嵌入式编程当中，面对资源占用紧张的情况下，难以驾驭众多的设计模式。

>它描述了在特定情境下如何组织类、对象及它们之间的交互从而解决常见的设计问题。设计模式不是可直接拷贝的代码，而是一种软件设计思想和可复用的代码结构设计方法

### 核心思想
1. 解耦：把硬件驱动和业务逻辑分离，降低修改风险和回归风险
2. 复用：把通用功能抽象成为可复用的模块，防止复制粘贴带来的维护负担
3. 通过接口和工厂机等机制新增功能的时候尽量不在动原来的代码，减少新功能引入的时候带来的bug概率。

### 单例模式：方便管理硬件资源

当一个mcu/soc 有很多的硬件资源需要管理的时候，把全部的逻辑写到同一个file里面显然是不合理的，这就出现了单例的模式，init的时候只需要单独的初始化一次即可。

__特点__:保证系统的某对象只有一个实例，用于管理硬件资源或者唯一资源
__适用场景__:USART，SPI，IIC,日志模块等唯一的资源
__优点__:避免重复初始化、配置冲突；提供统一访问点。
__注意事项__：避免把过多业务逻辑塞入单例，防止全局状态污染和测试困难.

``` cpp

// uart_singleton.c
#include <stdint.h>
#include <stdio.h>

typedef struct {
	int initialized;
	uint32_t baudrate;
} UART_Handle;

// 单独封装硬件初始化，避免业务层直接操作底层细节
static void UART_HwInit(UART_Handle *handle, uint32_t baudrate) {
	if (handle == NULL) {
		return;
	}

	handle->baudrate = baudrate;
	handle->initialized = 1;
	printf("UART initialized, baud=%u\n", baudrate);
}

UART_Handle* UART_GetInstance(void) {
	static UART_Handle uart = {0};

	// 首次调用时初始化，后续复用同一个实例
	if (!uart.initialized) {
		UART_HwInit(&uart, 115200);
	}

	return &uart;
}

```

### 工厂模式：简化外设初始化

> 当有很多个设备需要初始化的时候，工厂模式可以把初始化的逻辑抽象出来，提供一个统一的接口来创建对象。

``` cpp
// peripheral_factory.c
#include <stdint.h>
#include <stdio.h>

// 1) 外设类型：由工厂统一识别
typedef enum {
	DEVICE_UART = 0,
	DEVICE_SPI,
	DEVICE_I2C,
	DEVICE_MAX
} DeviceType;

// 2) 统一驱动接口：业务层只依赖这个抽象，不直接依赖具体外设实现
typedef struct {
	const char *name;
	void (*init)(void);
	void (*send)(const uint8_t *data, uint16_t len);
} DeviceDriver;

// 3) 各具体外设实现（这里用printf模拟硬件行为）
static void UART_Init(void) {
	printf("[UART] init, baud=115200\n");
}

static void UART_Send(const uint8_t *data, uint16_t len) {
	if (data == NULL || len == 0) {
		printf("[UART] invalid send param\n");
		return;
	}
	printf("[UART] send len=%u\n", len);
}

static void SPI_Init(void) {
	printf("[SPI] init, mode=0, clk=8MHz\n");
}

static void SPI_Send(const uint8_t *data, uint16_t len) {
	if (data == NULL || len == 0) {
		printf("[SPI] invalid send param\n");
		return;
	}
	printf("[SPI] send len=%u\n", len);
}

static void I2C_Init(void) {
	printf("[I2C] init, addr=0x50, speed=400kHz\n");
}

static void I2C_Send(const uint8_t *data, uint16_t len) {
	if (data == NULL || len == 0) {
		printf("[I2C] invalid send param\n");
		return;
	}
	printf("[I2C] send len=%u\n", len);
}

// 4) 具体产品对象（静态分配，避免嵌入式场景下动态内存碎片问题）
static const DeviceDriver uart_driver = {"UART", UART_Init, UART_Send};
static const DeviceDriver spi_driver  = {"SPI",  SPI_Init,  SPI_Send};
static const DeviceDriver i2c_driver  = {"I2C",  I2C_Init,  I2C_Send};

// 5) 工厂方法：根据类型返回对应驱动对象
static const DeviceDriver* DeviceFactory_Create(DeviceType type) {
	switch (type) {
		case DEVICE_UART: return &uart_driver;
		case DEVICE_SPI:  return &spi_driver;
		case DEVICE_I2C:  return &i2c_driver;
		default:          return NULL;
	}
}

// 6) 使用示例：业务层只拿抽象接口，不关心底层是UART/SPI/I2C
int main(void) {
	uint8_t tx_data[3] = {0xA1, 0xB2, 0xC3};

	const DeviceDriver *dev = DeviceFactory_Create(DEVICE_SPI);
	if (dev == NULL) {
		printf("Device create failed\n");
		return -1;
	}

	dev->init();
	dev->send(tx_data, sizeof(tx_data));

	return 0;
}

/*
为什么这样做：
1. 把“创建逻辑”收口到工厂，业务代码更干净。
2. 新增外设时，只需增加驱动实现 + 工厂分支，原业务代码不用改。
3. 嵌入式项目可用静态对象替代malloc，减少内存碎片风险。
*/

```

__特点__: 统一创建入口，屏蔽具体外设差异。
__适用场景__: 同一业务需要在不同通信外设间切换（UART/SPI/I2C）。
__优点__: 降低耦合、便于扩展、便于做板级差异适配。
__注意事项__: 工厂分支过多时可进一步改成“注册表工厂”避免switch膨胀。

### 适配器模式：兼容不同外设接口
有很多时候我们需要兼容不同厂商的外设，或者同一外设的不同版本，这时候适配器模式就非常有用，可以把不兼容的接口适配成统一的接口，或者有很多时候需要更换MCU/SOC，这时候适配器模式也可以把不同的硬件接口适配成统一的接口，方便业务层调用。

``` cpp
// sensor_adapter.c
#include <stdint.h>
#include <stdio.h>

// 厂商A传感器接口（历史接口）
static int VendorA_StartMeasure(uint8_t channel) {
	printf("VendorA start, channel=%u\n", channel);
	return 0;
}

// 厂商B传感器接口（新接口）
static int VendorB_Trigger(void) {
	printf("VendorB trigger\n");
	return 0;
}

// 统一目标接口：业务层只依赖这个抽象
typedef struct {
	int (*start)(void);
} SensorPort;

// 适配器A：把 VendorA 接口适配为统一接口
static int AdapterA_Start(void) {
	return VendorA_StartMeasure(1);
}

// 适配器B：把 VendorB 接口适配为统一接口
static int AdapterB_Start(void) {
	return VendorB_Trigger();
}

static const SensorPort sensor_a_port = { AdapterA_Start };
static const SensorPort sensor_b_port = { AdapterB_Start };

int main(void) {
	const SensorPort *sensor = &sensor_a_port;

	if (sensor->start() != 0) {
		printf("sensor start failed\n");
		return -1;
	}

	// 更换厂商时只切换适配器对象，业务逻辑不需要重写
	sensor = &sensor_b_port;
	if (sensor->start() != 0) {
		printf("sensor start failed\n");
		return -1;
	}

	return 0;
}

```






__特点__: 通过中间层把不兼容接口转换为统一接口。
__适用场景__: 替换MCU、替换驱动库、兼容历史代码。
__优点__: 新旧模块可以并存，迁移风险小。
__注意事项__: 适配层尽量只做“接口转换”，不要塞业务逻辑。


### 观察者模式：监控系统状态变化



观察者模式可以实现“事件发布-多模块订阅”，例如温度告警事件可以同时通知LED任务和日志任务。

``` cpp
// observer_event.c
#include <stdint.h>
#include <stdio.h>

#define MAX_OBSERVERS 4

typedef void (*EventCallback)(uint8_t event_id, uint32_t value);

typedef struct {
	EventCallback callbacks[MAX_OBSERVERS];
	uint8_t count;
} EventCenter;

static void EventCenter_Init(EventCenter *center) {
	if (center == NULL) {
		return;
	}
	center->count = 0;
}

static int EventCenter_Subscribe(EventCenter *center, EventCallback cb) {
	if (center == NULL || cb == NULL || center->count >= MAX_OBSERVERS) {
		return -1;
	}
	center->callbacks[center->count++] = cb;
	return 0;
}

static void EventCenter_Notify(EventCenter *center, uint8_t event_id, uint32_t value) {
	uint8_t i;
	if (center == NULL) {
		return;
	}
	for (i = 0; i < center->count; i++) {
		center->callbacks[i](event_id, value);
	}
}

static void LedTask_OnEvent(uint8_t event_id, uint32_t value) {
	printf("[LED] event=%u, value=%u\n", event_id, value);
}

static void LogTask_OnEvent(uint8_t event_id, uint32_t value) {
	printf("[LOG] event=%u, value=%u\n", event_id, value);
}

int main(void) {
	EventCenter center;

	EventCenter_Init(&center);
	EventCenter_Subscribe(&center, LedTask_OnEvent);
	EventCenter_Subscribe(&center, LogTask_OnEvent);

	// 模拟温度告警事件
	EventCenter_Notify(&center, 1, 85);
	return 0;
}

```

__特点__: 一对多通知，发布者与订阅者解耦。
__适用场景__: 传感器告警、状态广播、任务联动。
__优点__: 新增订阅者时不需要改发布者代码。
__注意事项__: 回调里不要做重阻塞操作，防止拖慢主流程。


### 对象池模式：管理有限资源实例

对象池模式可以避免频繁malloc/free导致的内存碎片，特别适合RTOS任务消息节点、连接对象、缓冲块管理。

``` cpp
// object_pool.c
#include <stdint.h>
#include <stdio.h>

#define POOL_SIZE 3

typedef struct {
	uint8_t id;
	uint8_t in_use;
} MsgNode;

typedef struct {
	MsgNode nodes[POOL_SIZE];
} MsgPool;

static void MsgPool_Init(MsgPool *pool) {
	uint8_t i;
	if (pool == NULL) {
		return;
	}
	for (i = 0; i < POOL_SIZE; i++) {
		pool->nodes[i].id = i;
		pool->nodes[i].in_use = 0;
	}
}

static MsgNode* MsgPool_Alloc(MsgPool *pool) {
	uint8_t i;
	if (pool == NULL) {
		return NULL;
	}
	for (i = 0; i < POOL_SIZE; i++) {
		if (!pool->nodes[i].in_use) {
			pool->nodes[i].in_use = 1;
			return &pool->nodes[i];
		}
	}
	return NULL;
}

static void MsgPool_Free(MsgNode *node) {
	if (node == NULL) {
		return;
	}
	node->in_use = 0;
}

int main(void) {
	MsgPool pool;
	MsgNode *n1;
	MsgNode *n2;

	MsgPool_Init(&pool);
	n1 = MsgPool_Alloc(&pool);
	n2 = MsgPool_Alloc(&pool);

	if (n1 != NULL) {
		printf("alloc node id=%u\n", n1->id);
	}
	if (n2 != NULL) {
		printf("alloc node id=%u\n", n2->id);
	}

	MsgPool_Free(n1);
	return 0;
}

```

__特点__: 启动时预分配，运行时复用。
__适用场景__: 高实时场景下的固定资源管理。
__优点__: 分配耗时稳定、避免堆碎片。
__注意事项__: 池容量要按峰值评估，避免资源耗尽。
